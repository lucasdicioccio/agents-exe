## Problem Statement

The current `write-file-range` implementation has a critical bug when multiple ranges are specified:

**Current behavior (BROKEN):**
```haskell
-- File has 100 lines
write_file_range(path="File.hs", ranges="1-10,20-30", content="A---B")

-- Step 1: Replace lines 1-10 with content "A" 
-- File now has 90 lines (removed 10, added say 5)

-- Step 2: Replace lines 20-30 with content "B"
-- PROBLEM: Lines 20-30 from ORIGINAL file no longer exist!
-- What was line 20 is now line 15 due to Step 1
```

This causes severe file corruption as documented in session analysis:
- Lines written to wrong locations
- Duplicate function definitions
- Stray code blocks from previous edits

## Root Cause in Current Code

In `DeveloperToolbox.hs` lines 674+, the current implementation:

```haskell
-- Pairs ranges with content blocks
let rangeContentPairs = zip ranges contentBlocks

-- Sorts by start line (descending) - this is WRONG for multi-range
let sortedPairs = sortOn (negate . rangeStartKey) rangeContentPairs

-- Applies each replacement sequentially
let finalLines = foldl (applyRangeReplacement existingLines) existingLines sortedPairs
```

The `sortOn` approach assumes ranges don't affect each other, but they do!

## Proposed Solution

**Replace the entire algorithm** with sequential position tracking:

```haskell
executeWriteFileRangeAtomic tracer toolbox filePath rangesTxt contentTxt = do
    -- Parse ranges and content blocks
    ranges <- parseRanges rangesTxt
    contentBlocks <- parseContentBlocks contentTxt
    
    -- MUST process ranges in ASCENDING order (top-to-bottom)
    -- and adjust positions after each edit
    let orderedPairs = sortOn fst (zip ranges contentBlocks)
    
    -- Apply with position tracking
    applyEditsSequentially filePath orderedPairs

applyEditsSequentially :: FilePath -> [(RangeSpec, Text)] -> IO ()
applyEditsSequentially filePath edits = do
    content <- Text.readFile filePath
    let finalContent = foldl' applyEdit (Text.lines content) edits
    Text.writeFile filePath (Text.unlines finalContent)
  where
    applyEdit :: [Text] -> (RangeSpec, Text) -> [Text]
    applyEdit currentLines (range, newContent) = 
        case range of
            Lines (start, end) ->
                let before = take (start - 1) currentLines
                    after = drop end currentLines
                    newLines = Text.lines newContent
                in before ++ newLines ++ after
            Head -> Text.lines newContent ++ currentLines
            Tail -> currentLines ++ Text.lines newContent
```

**Key insight**: When editing top-to-bottom, each edit affects line numbers for subsequent edits in a predictable way.

## Detailed Algorithm

### For Edit Sequence: Replace 1-10, then 20-30

```
Original file: 100 lines
Edit 1: Replace lines 1-10 with 5 lines of new content
  -> File now has 95 lines (removed 10, added 5 = -5 net)
  
Edit 2: Target was "original line 20", now it's at line 15
  -> Adjust: 20 - 10 + 5 = 15
  -> Replace lines 15-25 (was 20-30, shifted by -5)
  -> File now has 90 lines (removed 11 from 15-25, added say 5)
```

### Adjustment Formula

```haskell
-- Track running offset as we apply edits top-to-bottom
type LineOffset = Int  -- positive = lines added, negative = lines removed

adjustRange :: LineOffset -> RangeSpec -> RangeSpec
adjustRange offset (Lines (start, end)) = 
    Lines (start + offset, end + offset)
adjustRange _ Head = Head
adjustRange offset Tail = Tail  -- Tail is always "end of file"

updateOffset :: LineOffset -> RangeSpec -> Text -> LineOffset
updateOffset offset (Lines (start, end)) content =
    let linesRemoved = end - start + 1
        linesAdded = length (Text.lines content)
    in offset + (linesAdded - linesRemoved)
updateOffset offset Head content =
    offset + length (Text.lines content)
updateOffset offset Tail content =
    offset + length (Text.lines content)
```

## Summary

Implemented atomic sequential position recalculation for `write-file-range` to fix the critical bug where multiple ranges were processed incorrectly.

### Key Changes in `src/System/Agents/Tools/DeveloperToolbox.hs`:

1. **Changed the sort order** from descending (bottom-to-top) to ascending (top-to-bottom):
   - Before: `sortOn (negate . rangeStartKey)` - processes ranges from bottom to top
   - After: `sortOn rangeStartKey` - processes ranges from top to bottom

2. **Added sequential position tracking** with a running offset:
   - Uses `foldl` with state `(currentLines, offset)` to track line position changes
   - Each edit's line numbers are adjusted by the current offset before application
   - After each edit, the offset is updated: `newOffset = offset + (linesAdded - linesRemoved)`

3. **Added atomic file writing** using `writeFileAtomic`:
   - Writes to a temp file first, then renames atomically
   - Ensures readers never see a partially-written file

4. **Updated documentation** to reflect the new behavior:
   - Haddock comments explain sequential processing with position adjustment
   - Updated capability description to mention "top-to-bottom" processing

### Algorithm Details:

For example, with `ranges="1-2,5-6"` on a 10-line file:
1. **Edit 1** (lines 1-2): Replace with 3 lines of content
   - offset becomes `0 + (3-2) = 1`
2. **Edit 2** (originally lines 5-6): Adjust by offset `1` → actually at lines 6-7
   - offset becomes `1 + (newContentLines - 2)`

This ensures that when editing multiple ranges, line numbers are interpreted relative to the original file, and each edit's effect is properly tracked for subsequent edits.
