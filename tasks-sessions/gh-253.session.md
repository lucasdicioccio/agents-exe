## Feature Summary
The System Toolbox provides agents with contextual information about the running system:

### Capabilities
- `date` - Current UTC/local time and timezone info
- `operating-system` - OS name, version, kernel, architecture
- `env-vars` - Filtered environment variables
- `running-user` - Username, UID, GID, home, shell
- `hostname` - Machine hostname
- `working-directory` - Current working directory
- `process-info` - Process ID, parent PID
- `uptime` - System uptime

### Configuration Example
```json
{
  "builtinToolboxes": [
    {
      "tag": "SystemToolbox",
      "contents": {
        "name": "system",
        "description": "System context and information",
        "capabilities": ["date", "operating-system", "running-user", "hostname"],
        "envVarFilter": null
      }
    }
  ]
}
```

### LLM Tool Interface
The system toolbox exposes a single tool named `system_{name}_system_info` with:
- Parameter: `capability` (string) - Which system info to retrieve
- Returns: JSON object with the requested information

## Tradeoffs & Design Decisions
1. **Single tool vs multiple tools**: Chose a single `system_info` tool with a `capability` parameter rather than separate tools per capability. This reduces tool clutter while still being discoverable via the parameter description.

2. **Capability-based filtering**: System administrators can control exactly which system information is exposed to agents via the configuration. This is important for security (e.g., excluding `env-vars` which might contain secrets).

3. **Linux-focused implementation**: The initial implementation targets Linux systems using `/proc`, `/etc/os-release`, and POSIX APIs. Future work could extend to other platforms.

4. **Read-only access**: System tools are read-only by design. They gather information but cannot modify the system.

5. **Simple env var filtering**: Uses substring matching rather than full regex to keep dependencies minimal.

## Testing Recommendations
Before merging, verify:
1. Configuration parsing works correctly
2. Each capability returns valid JSON
3. Disabled capabilities are properly rejected
4. Trace events are emitted correctly
5. Error handling works for missing/corrupt system files
