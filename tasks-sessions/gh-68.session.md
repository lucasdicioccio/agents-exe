# Design Export/Import Data Types and Package Format

## Overview
Design the core data types and serialization format for exporting/importing agent configurations, tools, and MCP server configurations.

## Background
From `System.Agents.Base`:
```haskell
data Agent = Agent
    { slug :: Text
    , apiKeyId :: Text
    , flavor :: Text
    , modelUrl :: Text
    , modelName :: Text
    , announce :: Text
    , systemPrompt :: [Text]
    , toolDirectory :: FilePath
    , mcpServers :: Maybe [McpServerDescription]
    }
```

Tools are stored in `toolDirectory` as executable scripts following the `describe|run` protocol (see `System.Agents.Tools.Bash`).

## Requirements

### 1. Create Export Package Types
Define a type to represent a complete exportable package:

```haskell
data ExportPackage = ExportPackage
    { packageMetadata :: PackageMetadata
    , packageAgents :: [AgentExport]
    , packageTools :: [ToolExport]  -- standalone tools, not attached to agents
    , packageMcpServers :: [McpServerExport]
    }

data AgentExport = AgentExport
    { agentConfig :: Agent
    , agentNamespace :: Maybe Text  -- for git-based multi-agent repos
    , agentTools :: [ToolExport]    -- tools embedded with this agent
    }

data ToolExport = ToolExport
    { toolName :: Text
    , toolContent :: ByteString     -- the executable script content
    , toolPermissions :: FileMode   -- executable permissions
    , toolNamespace :: Maybe Text   -- for namespacing in git repos
    }

data McpServerExport = McpServerExport
    { mcpConfig :: McpServerDescription
    , mcpNamespace :: Maybe Text
    }

data PackageMetadata = PackageMetadata
    { packageVersion :: Text        -- schema version for compatibility
    , packageCreatedAt :: UTCTime
    , packageDescription :: Maybe Text
    , packageSource :: Maybe Text   -- git URL or origin info
    }
```

### 2. Define Namespace Support
Namespaces allow multiple agents/tools in a single git repository:

```haskell
-- | A namespace path like "team1/project-a/agent-name"
newtype Namespace = Namespace { unNamespace :: [Text] }
    deriving (Show, Eq, Ord)

-- | Parse a namespace from dot-notation or path-notation
parseNamespace :: Text -> Either String Namespace
-- | Convert namespace to filesystem path
namespaceToPath :: Namespace -> FilePath
-- | Convert namespace to git-style path
namespaceToGitPath :: Namespace -> FilePath
```

### 3. JSON Serialization
Implement `ToJSON`/`FromJSON` for all export types with schema versioning:
- Version 1.0.0: Initial format supporting agents, tools, MCP servers
- Include migration logic for future schema changes

### 4. Files to Create/Modify
- **Create**: `src/System/Agents/ExportImport/Types.hs` - Core data types
- **Create**: `src/System/Agents/ExportImport/Namespace.hs` - Namespace handling
- **Create**: `src/System/Agents/ExportImport/JSON.hs` - Serialization

### 5. Key Design Decisions

**Trade-off: Embedded vs. Referenced Tools**
- Option A: Embed tool content in JSON (self-contained, larger files)
- Option B: Reference tools by path (smaller files, requires directory structure)
- **Decision**: Support both modes - embed for single-file exports, reference for git repos

**Trade-off: Flat vs. Hierarchical Namespaces**
- Flat: `team1.agent-name` (simpler, limited nesting)
- Hierarchical: `team1/project-a/agent-name` (more expressive)
- **Decision**: Support hierarchical with dot-notation shorthand
