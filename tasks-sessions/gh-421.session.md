### 💬 User Query

# Network endpoint Configuration: generalize the header/token configurations

# Context

Today one can add HTTP headers and tokens to network endpoints such as OpenAPI and PostgREST endpoints.
A problem we may face is that token are sensitive and often time-bound.
Another problem we may face is that headers sometimes are non-standard for operational reasons (e.g., a custom x-apikey header).

As a result:

  -  (need-1) we should allow configurations to read from places like an env-var or from a file or from a secret manager like pass
  -  (need-2) we should allow configurations to define where in the http-request they are serialized (i.e., which header and possibly with wich format)

# Task

Implement a solution to both need-1 and need-2 in a single PR.

# Design

Revise the configuration of HTTP endpoint configurations, by adding a "Secrets" json configuration key (we keep Token and Headers, but we add a new mechanism).

Secrets is a list of Secret. A Secret is defined with:

-    a source, which is a sum type with possible values (Given Text) | (ApiKey KeyId) | (FileSystem FilePath) | (EnvVar Text) | (Command Name [Arg]) -- here ApiKey refers to the same file as the LLM completion ApiKeys
-    a decoder with possible values Clear (Bool for trimming or not final newline) | Base64 | Hexadecimal
-    a serialize with possible values Header (Name) (Maybe Template) | QueryString Name -- here Template allows to express simple replacements like Bearer {{secret}}

Then, secret are read at toolbox initialization.
 Add some TODO note for later to consider the risk of logging secrets in non-standard serialize locations.

## Summary

I have successfully implemented a flexible secrets management system for HTTP endpoint configurations (OpenAPI and PostgREST toolboxes) that addresses both requirements from the issue:

### Implementation Details

**1. New Module: `System.Agents.Tools.Secrets`**
- **SecretSource**: Defines where secrets come from:
  - `Given Text` - Inline value
  - `ApiKeySource Text` - Reference to API key file (same as LLM keys)
  - `FileSystem FilePath` - Read from file (e.g., Docker secrets, systemd credentials)
  - `EnvVar Text` - Read from environment variable
  - `Command Text [Text]` - Execute command and capture stdout

- **SecretDecoder**: How to decode secrets:
  - `Clear Bool` - Plain text, with optional newline trimming
  - `Base64` - Base64 encoded
  - `Hexadecimal` - Hexadecimal encoded

- **SecretSerializer**: Where to place secrets in HTTP requests:
  - `Header Text (Maybe Template)` - HTTP header with optional template (e.g., `Bearer {{secret}}`)
  - `QueryString Text` - Query string parameter

- **Secret Resolution**: Secrets are resolved at toolbox initialization time via `resolveSecrets` and `resolveSecret` functions.

**2. Updated Configuration Types (`System.Agents.Base`)**
- Added `openApiSecrets :: Maybe [Secret]` to `OpenAPIServerDescription`
- Added `postgrestSecrets :: Maybe [Secret]` to `PostgRESTServerDescription`

**3. Updated Toolbox Initialization (`OpenAPIToolbox.hs` and `PostgRESToolbox.hs`)**
- Both `initializeToolbox` functions now accept an API keys file path
- Secrets are resolved at initialization and stored in the Toolbox state
- Resolved secrets are applied to all outgoing HTTP requests via `applySecretsToHeaders` and `applySecretsToQueryString`

**4. Updated Tool Loading (`AgentTree.ToolLoader` and `AgentTree`)**
- `loadAgentTools` now accepts the API keys file path and passes it through to toolbox initialization
