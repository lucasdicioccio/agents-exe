# Add request body support for PostgREST toolbox POST/PUT/PATCH operations

Depends-on:

## Summary

The PostgREST toolbox currently supports GET operations with filters, pagination, and ordering, but **does not expose the `body` parameter for POST/PUT/PATCH operations**. This prevents users from inserting or updating rows in the database.

## Problem

When using the PostgREST toolbox to perform write operations (POST, PUT, PATCH), there is no way to provide the request body containing the data to insert or update. The `ToolParameters` type already has `tpRequestBody :: Maybe Schema` populated from the OpenAPI spec, but `buildPostgRESTParamProperties` in `ToolRegistration.hs` doesn't include it in the LLM tool schema.

## Solution

Modified `buildPostgRESTParamProperties` in `src/System/Agents/ToolRegistration.hs` to:

1. **Extract the request body schema** from `tpRequestBody` when available
2. **Build a `body` parameter property** with appropriate type:
   - `ObjectParamType` for single row inserts (schema type = "object")
   - `ObjectParamType` for bulk inserts (schema type = "array")
3. **Include schema properties** from the OpenAPI spec for better LLM understanding
4. **Generate helpful descriptions** indicating how to provide the body

## Changes Made

### `src/System/Agents/ToolRegistration.hs`

Added new helper functions to `buildPostgRESTParamProperties`:

- `buildBodyParamType` - Converts OpenAPI Schema to ParamType
- `buildBodyDescription` - Generates description with type hints
- `buildSchemaProperties` - Extracts object properties from schema
- `schemaToParamProperty` - Converts schema properties to ParamProperty
- `schemaTypeToParamType` - Maps JSON Schema types to ParamType

The `body` parameter is:
- **Optional** (`propertyRequired = False`) - allowing inserts with defaults
- **Typed** according to the OpenAPI schema
- **Documented** with description from the spec

## Example Usage

After this change, users can insert items like:

```json
{
  "body": {
    "name": "Max Revive",
    "description": "Fully restores HP",
    "category": "revive",
    "price": 4000,
    "effect": {"revive": true, "hp_percent": 100},
    "is_usable_in_battle": true,
    "is_consumable": true
  }
}
```

Or bulk insert:

```json
{
  "body": [
    {"name": "Item 1", "price": 100},
    {"name": "Item 2", "price": 200}
  ]
}
```

## Tradeoffs

1. **Optional vs Required Body**: The body is marked as optional to allow inserting rows with database defaults. This is more flexible but means the LLM might omit the body for required fields, resulting in a database error.

2. **Schema Complexity**: Complex nested schemas are simplified to `OpaqueParamType` for deeply nested structures. This provides flexibility but less validation.

3. **Type Inference**: The `OpaqueParamType "object"` fallback allows any JSON structure, which is necessary for flexibility but provides less type safety than strict schema validation.
