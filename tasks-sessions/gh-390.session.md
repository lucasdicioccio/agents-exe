   - New constructor `SqliteSnapshot` for creating per-conversation database copies
   - Updated JSON serialization to support `"snapshot"` as a valid access mode value

2. **System.Agents.Tools.SqliteToolbox** - Major updates:
   - Added `Snapshot` to the internal `AccessMode` type
   - Added `SnapshotState` type to track lazy snapshot initialization
   - Added `SnapshotError` to `QueryError` for snapshot-specific errors
   - Added `SnapshotCreatedTrace` and `SnapshotErrorTrace` to `Trace` for monitoring
   - Modified `Toolbox` to support optional connections for snapshot mode
   - Added `executeQueryWithContext` function that accepts an optional `ConversationId`
   - Snapshot creation is lazy - happens on first query execution with a conversation context
   - Snapshot file naming: `original.{uuid}.snapshot.sqlite` where uuid is the conversation ID

3. **System.Agents.ToolRegistration** - Updated to pass conversation context:
   - Modified `sqliteTool` to extract `ctxConversationId` from the tool execution context
   - The conversation ID is now passed to `executeQueryWithContext` for snapshot mode support

### How it works:

1. When a toolbox is configured with `"access": "snapshot"` in the agent configuration
2. The toolbox initializes without creating a database connection
3. On first query, the toolbox:
   - Creates a copy of the original database with the conversation ID as suffix
   - Opens the copy in read-write mode
   - Executes the query on the copy
4. Subsequent queries in the same conversation reuse the same snapshot
5. Changes are isolated to the conversation's snapshot and don't affect the original database

### Example configuration:

```json
{
  "tag": "SqliteToolbox",
  "contents": {
    "name": "sandbox-db",
    "description": "Sandbox database for isolated changes",
    "path": "/path/to/original.db",
    "access": "snapshot"
  }
}
```
