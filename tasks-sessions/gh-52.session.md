### User Query

# Extract and display LLM thinking/reasoning content separately

## Summary

Modern LLMs like OpenAI's o1/o3 series and Claude with extended thinking provide a separate "reasoning" or "thinking" content in their responses. The codebase already parses this field (`rspReasoningContent` in `OpenAI.Response`) but it's not being stored in the session or displayed to users.

This issue tracks extracting the thinking content and displaying it properly in both the TUI Conversation view and the SessionPrint command output.

## Background

The `System.Agents.LLMs.OpenAI` module already parses `reasoning_content` from API responses:

```haskell
data Response = Response
    { ...
    , rspReasoningContent :: Maybe Text  -- Already parsed!
    }
```

However, this is currently discarded in `System.Agents.Session.OpenAI.extractResponse`:

```haskell
extractResponse rawValue rsp =
    let llmRsp = LlmResponse rsp.rspContent rawValue  -- thinking content lost!
```

## Required Changes

### 1. Core Data Type (`System.Agents.Session.Base`)

Add a field to `LlmResponse` to store thinking content:

```haskell
data LlmResponse = LlmResponse 
    { responseText :: Maybe Text
    , responseThinking :: Maybe Text  -- NEW: separate thinking/reasoning content
    , rawResponse :: Aeson.Value
    }
```

**Note:** This is a breaking change to the `LlmResponse` constructor. Update pattern matches in:
- `System.Agents.AgentTree.OneShotTool.extractResponseText`
- `System.Agents.MCP.Server.extractResponseText`  
- `System.Agents.OneShot.extractResponseText`

### 2. Response Extraction (`System.Agents.Session.OpenAI`)

Update `extractResponse` to populate the new field:

```haskell
extractResponse rawValue rsp =
    let llmRsp = LlmResponse 
            { responseText = rsp.rspContent
            , responseThinking = rsp.rspReasoningContent  -- NEW
            , rawResponse = rawValue
            }
```

### 3. SessionPrint Display (`System.Agents.SessionPrint`)

Update `formatLlmTurn` to display thinking content before the main response:

```haskell
formatLlmTurn :: SessionPrintOptions -> Session.LlmTurnContent -> Text.Text
formatLlmTurn _opts content =
    let thinkingSection = case content.llmResponse.responseThinking of
            Just thinking -> "### Thinking Process\n\n<details>\n<summary>Click to expand reasoning</summary>\n\n" <> thinking <> "\n</details>\n\n"
            Nothing -> ""
        responseSection = ...
    in thinkingSection <> responseSection <> toolCallsSection
```

**Tradeoffs for display style:**
- Option A: Collapsible `<details>` HTML block (shown above) - keeps output concise by default
- Option B: Separate markdown section - always visible
- Option C: Blockquote with label - simple but may be verbose

I recommend **Option A** with a `<details>` element for SessionPrint since:
- Thinking content can be very verbose
- Users typically want to see the final response first
- Markdown renderers support HTML details elements

### 4. TUI Display (`System.Agents.TUI.Render`)

Update `render_turn` for `LlmTurn` to show thinking content:

```haskell
render_turn (k, turn) =
    case turn of
        LlmTurn llmTurn ->
            withAttr llmMessageAttr $
                vBox
                    [ -- NEW: thinking section
                      case llmTurn.llmResponse.responseThinking of
                          Just thinking -> 
                              withAttr thinkingAttr $
                                  vBox [ txt "🤔 Thinking...", txt thinking, txt "" ]
                          Nothing -> emptyWidget
                    , case llmTurn.llmResponse.responseText of ...
```

Also add a new attribute for thinking content:

```haskell
-- In tui_appAttrMap
, (thinkingAttr, BrickUtil.fg Vty.magenta `Vty.withStyle` Vty.italic)
```

**Tradeoffs for TUI display:**
- Option A: Show thinking inline (as shown) - simple but takes screen space
- Option B: Collapsible with key binding - more complex UI but saves space
- Option C: Separate "show thinking" toggle mode - requires state changes

I recommend **Option A** for simplicity, with a distinct color/style to differentiate thinking from the final response.

## Data Migration Considerations

Existing session JSON files will have `LlmResponse` objects without the `responseThinking` field. The `FromJSON` instance should handle this gracefully by defaulting to `Nothing` when the field is missing (standard aeson behavior for `Maybe` fields).
