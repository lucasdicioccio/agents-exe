# DEV: Add patch-file capability (unified diff format) to DeveloperToolbox

Depends-on:

## Problem Statement

The current `write-file-range` capability has fundamental limitations when editing multiple locations in a file:

1. **Line number fragility**: When editing multiple ranges, each edit shifts line numbers for subsequent edits, causing the LLM to target wrong locations
2. **No validation before application**: Invalid edits partially corrupt files, requiring git revert
3. **Poor conflict detection**: Cannot detect when multiple edits would conflict with each other

Evidence from sessions shows this causes significant file corruption:
- Session `e3b72fbf`: "stray case expression (lines 666-677) from different function"
- Session `706c7c9d`: "Lines 1013-1019 contain code that doesn't belong in handleClipboardPaste"
- Session `b2c76d12`: Documented corruption types including "Duplicate imports/lines, Misplaced code blocks"

## Proposed Solution

Add a new `patch-file` capability that accepts **unified diff format** patches:

```haskell
-- New capability
DevToolPatchFile :: DeveloperToolCapability

executePatchFile ::
    Tracer IO Trace ->
    Toolbox ->
    FilePath ->      -- ^ Path to the file to patch
    Text ->          -- ^ Unified diff patch content
    IO (Either DeveloperToolError PatchResult)
```

### Example Usage

```haskell
-- LLM generates a standard unified diff
patchContent = """
--- a/src/System/Agents/TUI/Event.hs
+++ b/src/System/Agents/TUI/Event.hs
@@ -50,0 +51 @@ import Foo
+import Data.Text (Text)
@@ -100,5 +101,5 @@ func1 x =
-  oldBody
+  newBody
@@ -200,3 +201,4 @@ func2 y =
   line1
   line2
+  insertedLine
   line3
"""
```

### Benefits of Diff Format

1. **Context-aware**: Each hunk includes context lines (e.g., `@@ -100,5 +101,5 @@`) enabling validation before application
2. **Atomic application**: Validate all hunks first, then apply - no partial corruption
3. **Standard tooling**: Can use `Diff` library or shell out to `patch` command
4. **Human-readable**: Easy to debug and review

## Implementation Details

### Data Types

```haskell
-- | Result of a patch operation
data PatchResult = PatchResult
    { patchFilePath :: FilePath
    , patchHunksApplied :: Int
    , patchHunksRejected :: Int
    , patchLinesChanged :: Int
    }
    deriving (Show)

-- | New trace events
data Trace = ...
    | PatchFileStartedTrace !FilePath !Int  -- ^ path, hunk count
    | PatchFileCompletedTrace !FilePath !Int !Int  -- ^ applied, rejected
    | PatchFileErrorTrace !Text !Text
```

### Algorithm

```haskell
executePatchFile tracer toolbox filePath patchText = do
    -- 1. Parse the unified diff into hunks
    hunks <- parseUnifiedDiff patchText
    
    -- 2. Read the current file content
    content <- Text.readFile filePath
    let lines = Text.lines content
    
    -- 3. Validate ALL hunks before applying any
    validated <- mapM (validateHunk lines) hunks
    
    -- 4. If any hunk fails validation, return error (atomic)
    case sequence validated of
        Left err -> return $ Left $ PatchValidationError err
        Right validHunks -> do
            -- 5. Apply hunks bottom-to-top (descending line order)
            let newLines = foldr applyHunk lines (reverse validHunks)
            Text.writeFile filePath (Text.unlines newLines)
            return $ Right $ PatchResult ...
```

### Validation Rules

Each hunk must:
1. Match context lines exactly (fuzz factor optional, default 0)
2. Have line numbers within valid range
3. Not overlap with other hunks (unless explicitly allowed)

### Error Types

```haskell
data PatchError
    = ParseError !Text           -- ^ Invalid diff format
    | ContextMismatch !Int !Text -- ^ Line N context doesn't match
    | HunkOverlap !Int !Int      -- ^ Hunks at lines X and Y overlap
    | FileNotFound !FilePath
```
