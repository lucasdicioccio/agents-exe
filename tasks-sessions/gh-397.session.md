# Add validate-agent and create-agent/tool capabilities to DeveloperToolbox

Depends-on: 
Base-branch: main

## Overview

Extend the `DeveloperToolbox` with two new capabilities:

1. **`validate-agent`** - Read and validate an agent JSON file against the schema
2. **`create-agent` / `create-tool`** - Write new agent/tool configurations from first principles, with optional reference from already-validated existing agents/tools

## Background

The `DeveloperToolbox` (defined in `System.Agents.Tools.DeveloperToolbox`) currently provides:
- `DevToolValidateTool` - Validates bash tool scripts
- `DevToolScaffoldAgent` - Generates agent scaffolding from templates
- `DevToolScaffoldTool` - Generates tool scaffolding in various languages
- `DevToolShowSpec` - Shows specification documentation

The existing scaffolding tools create minimal templates. This enhancement adds **validation** for agent files and **creation from reference** capabilities to enable more sophisticated agent/tool development workflows.

## Detailed Requirements

### 1. `validate-agent` Capability

**New Capability:** `DevToolValidateAgent`

**Purpose:** Read and validate an agent JSON configuration file, returning detailed validation results.

**Validation Checks:**
- JSON is valid and parseable
- Required fields are present (`slug`, `apiKeyId`, `flavor`, `modelUrl`, `modelName`, `announce`, `systemPrompt`)
- Field types are correct
- Tool references are valid (optional - could be expanded later)
- Extra agent references can be resolved (optional)
- No circular references in agent tree (optional)

## Usage Examples

### Validating an Agent
```json
{
  "tool": "developer_validate_agent",
  "arguments": {
    "path": "agents/my-agent.json"
  }
}
```

Response:
```json
{
  "path": "agents/my-agent.json",
  "valid": true,
  "slug": "my-agent",
  "errors": [],
  "warnings": ["No tool sources configured"]
}
```

### Creating Agent from Scratch
```json
{
  "tool": "developer_create_agent",
  "arguments": {
    "output_path": "agents/new-agent.json",
    "overrides": {
      "slug": "new-agent",
      "model_name": "gpt-4",
      "system_prompt": ["You are a helpful assistant."]
    }
  }
}
```

### Creating Agent from Reference
```json
{
  "tool": "developer_create_agent",
  "arguments": {
    "reference_path": "agents/template.json",
    "output_path": "agents/customized.json",
    "overrides": {
      "slug": "customized",
      "model_name": "gpt-3.5-turbo"
    }
  }
}
```

### Creating Tool from Reference
```json
{
  "tool": "developer_create_tool",
  "arguments": {
    "language": "bash",
    "reference_path": "tools/existing-tool.sh",
    "output_path": "tools/new-tool.sh",
    "config": {
      "slug": "new-tool",
      "description": "A modified version of existing tool"
    }
  }
}
```

## Testing Considerations

1. **Unit Tests:**
   - Valid agent file passes validation
   - Invalid JSON reports parse error
   - Missing required fields reports specific errors
   - Reference mode correctly merges overrides

2. **Integration Tests:**
   - Create agent → validate agent round-trip
   - Create tool → validate tool round-trip
   - Reference existing agent/tool and verify inheritance

3. **Edge Cases:**
   - Empty override object (exact copy)
   - All fields overridden (complete replacement)
   - Non-existent reference file (error handling)
   - File already exists without force flag (error)

## Backward Compatibility

- New capabilities are opt-in via `developerToolboxCapabilities` list
- Existing capabilities remain unchanged
- New trace events extend (don't replace) existing trace type
- New `CallResult` constructors extend existing type

## Tradeoffs

1. **Complexity vs. Power:** The `AgentOverrides` type is large (one field per Agent field) but provides complete control. Alternative: use a simpler key-value approach with less type safety.

2. **Validation Strictness:** Should validation check tool path existence? (Probably not - tools may be created after agent.)

3. **Reference Mode Semantics:** Should we support deep merging of nested structures (e.g., `bashToolboxes`) or just top-level replacement? Recommendation: start with top-level replacement for simplicity.

4. **Language Support for Tools:** Only bash tools can be loaded for reference. Python/Haskell tools would require parsing their description output. This is acceptable limitation for v1.

## Future Enhancements (Out of Scope)

- Validate agent tree for circular references
- Support creating MCP/OpenAPI/PostgREST toolbox configs
- Interactive mode for guided agent creation
- Template library with common agent patterns
- Diff mode: show changes between reference and created agent
