# Capture nested sub-agent calls from ToolPortal for TUI visibility

Depends-on: 
Base-branch: main
Final: false

## Problem

When a Lua toolbox script calls a sub-agent via the ToolPortal (using `tools.call("prompt_agent_xxx", {...})`), the nested conversation is **not captured and displayed in the TUI**. This works correctly when an agent directly calls a sub-agent, but fails when the sub-agent call originates from within a Lua script.

### Root Cause Analysis

The issue stems from context propagation through the ToolPortal:

1. **Direct agent subcall (works):** When `turnAgentRuntimeIntoIOTool` is invoked directly, it receives a `ToolExecutionContext` with `ctxWorld` and `ctxEventQueue` populated. It uses these to emit `OSEvent_SubcallStarted`, `OSEvent_SubcallProgress`, etc., which the TUI captures.

2. **Lua portal subcall (broken):** When Lua calls `tools.call()`, the `ToolPortal.executeTool` function creates a **new minimal context** using `mkMinimalContext`:
   ```haskell
   let minimalCtx = mkMinimalContext sessId convId turnId portal
   ```
   This minimal context has:
   - `ctxWorld = Nothing`
   - `ctxEventQueue = Nothing`
   
   When the oneshot agent tool runs, it checks `ctxWorld` and `ctxEventQueue` to emit events, but since they're `Nothing`, no events are emitted. The TUI never learns about the subcall.

### Code Locations

**File: `src/System/Agents/ToolPortal.hs`**
```haskell
executeTool :: ...
executeTool tracer portal tool args = do
    -- Creates NEW session/conversation IDs - loses parent context!
    sessId <- newSessionId
    convId <- newConversationId
    turnId <- newTurnId
    let minimalCtx = mkMinimalContext sessId convId turnId portal  -- No World/EventQueue!
    ...
```

**File: `src/System/Agents/AgentTree/OneShotTool.hs`**
```haskell
runSubAgent ctx (PromptOtherAgent query) = do
    -- Extract OS integration fields from context
    let mWorld = Ctx.ctxWorld ctx           -- Always Nothing for portal calls
    let mEventQueue = Ctx.ctxEventQueue ctx -- Always Nothing for portal calls
    
    case mWorld of
        Just world -> do  -- This branch never executes for portal calls
            -- Insert conversation into OS World
        Nothing -> pure ()  -- Always this branch
    
    case mEventQueue of
        Just eventQueue -> do  -- This branch never executes for portal calls
            -- Emit OSEvent_SubcallStarted
        Nothing -> pure ()  -- Always this branch
```

### Expected Behavior

When Lua calls `tools.call("prompt_agent_foo", {what = "..."})`:
1. The subcall should appear as a nested conversation in the TUI
2. Progress events should update the conversation as it runs
3. Completion/failure events should mark it as done
4. Parent-child relationships should be preserved for tree visualization

### Reproduction Steps

1. Start the TUI with an agent that has Lua tools enabled
2. Create a Lua script that calls a sub-agent:
   ```lua
   local tools = require("tools")
   local result = tools.call("prompt_agent_architect", {what = "Design a module"})
   return result
   ```
3. Execute the Lua script
4. **Bug:** The architect subcall does not appear as a nested conversation in the TUI

### Impact

- **TUI usability:** Users cannot see nested agent calls initiated from Lua scripts
- **Debugging difficulty:** Subcall conversations are invisible, making it hard to trace issues
- **Inconsistent experience:** Direct agent calls show nesting, Lua-initiated calls don't

### Tradeoffs to Consider

1. **Context propagation complexity:** The ToolPortal currently creates fresh contexts to prevent context leakage. We need to selectively propagate only the OS integration fields.

2. **Session/Conversation ID generation:** `executeTool` currently generates fresh IDs. For proper nesting, these should either:
   - Be passed from the parent context, OR
   - Be linked via the existing `ctxParentConversation` mechanism

3. **Portal recursion safety:** The minimal context was designed to prevent infinite loops. We must ensure that propagating World/EventQueue doesn't break this safety.

### Proposed Solution

Modify `ToolPortal.executeTool` to accept and propagate the parent context's OS integration fields:

```haskell
-- Option 1: Pass full context and extract what we need
executeTool :: ...
            -> ToolExecutionContext  -- Parent context
            -> IO (Either ...)
executeTool tracer portal parentCtx tool args = do
    -- Generate new IDs for this tool execution
    sessId <- newSessionId
    convId <- newConversationId
    turnId <- newTurnId
    
    -- Create minimal context but preserve OS fields from parent
    let minimalCtx = (mkMinimalContext sessId convId turnId portal)
            { ctxWorld = ctxWorld parentCtx
            , ctxEventQueue = ctxEventQueue parentCtx
            , ctxParentConversation = Just (ctxConversationId parentCtx)
            }
    ...
```

This would require modifying the `ToolPortal` type to accept the parent context, or the tool execution mechanism to pass context through.
