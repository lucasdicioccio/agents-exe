# Implement `toolbox nest` command to wrap multiple tools into a single nested tool

## Overview

Implement `agents-exe toolbox nest` command that maps an agent's toolbox into a single nested tool. This allows exposing multiple tools through a single bash-tool-compatible interface, making it useful for directly testing MCP tools with the simpler bash-tool interface.

## Motivation

When working with MCP tools, testing and debugging can be cumbersome because MCP requires a full protocol implementation. By nesting tools behind a bash-compatible interface, users can:

1. Test MCP tools using simple bash scripts
2. Create composite tools that delegate to multiple underlying tools
3. Use tools in environments that only support bash-style tool invocation

This implements the pattern described in docs/internal-tool-mapping.md.

## Proposed Interface

```bash
# Nest all tools from an agent into a single tool
agents-exe toolbox nest --agent-file agent.json --output nested-tool.sh

# Nest with a specific name
agents-exe toolbox nest --agent-file agent.json --name my-nested-tool --output my-tool.sh

# Describe the nested tool (shows available sub-commands)
./my-tool.sh describe

# Run a nested tool
./my-tool.sh run --tool bash_hello --arg '{"name":"world"}'
```

## Nested Tool Protocol

The generated nested tool implements the standard bash-tool interface:

### `describe` command
Returns a `ScriptInfo` that describes the nested tool as a single tool:

```json
{
  "slug": "my-nested-tool",
  "description": "A nested tool providing access to 5 sub-tools",
  "args": [
    {
      "name": "tool",
      "description": "Name of the sub-tool to invoke",
      "type": "string",
      "backing_type": "string",
      "arity": "single",
      "mode": "dashdashequal"
    },
    {
      "name": "arg",
      "description": "JSON arguments for the sub-tool",
      "type": "string",
      "backing_type": "string",
      "arity": "single",
      "mode": "dashdashequal"
    }
  ]
}
```

### `run` command
Invokes the specified sub-tool with the provided arguments:

```bash
./my-tool.sh run --tool=bash_hello --arg='{"name":"world"}'
```

### Generated Script Structure

The generated script is a self-contained bash script that:

1. Embeds the nest configuration as a JSON blob
2. Implements the `describe` command
3. Implements the `run` command that:
   - Parses the `--tool` and `--arg` parameters
   - Looks up the tool in the embedded config
   - Dispatches to the appropriate handler based on tool type

```bash
#!/bin/bash
# Generated nested tool: my-nested-tool
# This script was generated by agents-exe toolbox nest

NEST_CONFIG='{"name":"my-nested-tool","tools":[...]}'

case "$1" in
  describe)
    echo '{"slug":"my-nested-tool",...}'
    ;;
  run)
    # Parse arguments and dispatch
    ...
    ;;
  *)
    echo "Unknown command: $1"
    exit 1
    ;;
esac
```

## Tradeoffs

### Pros
- Allows testing MCP tools with bash-compatible interfaces
- Self-contained scripts can be distributed independently
- Enables composition of multiple tools into a single interface

### Cons
- Adds a layer of indirection (tool → nested tool → actual tool)
- MCP tools may still require agents-exe to be installed for full protocol support
- Generated scripts embed configuration that can become stale
- Some tool features may not translate well through the nesting layer

## Alternative Approaches Considered

1. **Dynamic loading**: Instead of generating a static script, create a long-running process. This was rejected because it loses the simplicity of the bash-tool interface.

2. **Pure bash implementation**: Implement all dispatch logic in bash without calling back to agents-exe. This was rejected because MCP protocol handling is too complex for bash.

3. **Embed agents-exe**: The generated script calls back to agents-exe for complex operations. This provides a balance between simplicity and functionality.

## Testing Considerations

2. Test with MCP tools (requires agents-exe callback)
3. Test with mixed tool types
4. Verify generated script has correct permissions
5. Test argument passing and JSON encoding
6. Test error handling when sub-tool fails
