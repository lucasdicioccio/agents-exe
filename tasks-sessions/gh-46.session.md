# Add ToolExecutionContext to Tool type and update tool runners

## Overview

Modify the `Tool` type and all tool execution paths to accept `ToolExecutionContext`. This is an internal change - the LLM-facing interface remains unchanged.

## Current State

The `Tool` type currently takes an `rtVal` parameter (which is `ConversationId`):

```haskell
-- In System.Agents.Tools
data Tool rtVal call = Tool
    { toolDef :: ToolDef
    , toolRun :: Tracer IO ToolTrace -> rtVal -> Aeson.Value -> IO (CallResult call)
    }

-- Current usage in bashTool:
toolRun :: Tracer IO ToolTrace -> a -> Aeson.Value -> IO (CallResult ())
-- 'a' is ignored for bash tools
```

## Required Changes

### 1. Update Tool type (`System.Agents.Tools`)

Replace the generic `rtVal` with `ToolExecutionContext`:

```haskell
data Tool call = Tool
    { toolDef :: ToolDef
    , toolRun :: Tracer IO ToolTrace -> ToolExecutionContext -> Aeson.Value -> IO (CallResult call)
    }
```

### 2. Update ToolRegistration (`System.Agents.ToolRegistration`)

```haskell
data ToolRegistration = ToolRegistration
    { innerTool :: Tool ()  -- no longer needs rtVal parameter
    , declareTool :: OpenAI.Tool
    , findTool :: OpenAI.ToolCall -> Maybe (Tool OpenAI.ToolCall)
    }
```

### 3. Update all tool constructors

- `bashTool` - pass context via env vars or special args (see separate issue)
- `ioTool` - pass context to IO action
- `mcpTool` - pass context (may need MCP protocol consideration)

### 4. Update call sites

Update `executeToolCall` in `System.Agents.OneShot` and `System.Agents.MCP.Server` to construct and pass `ToolExecutionContext`.

## Files to Modify

- `src/System/Agents/Tools.hs` - Tool type and constructors
- `src/System/Agents/ToolRegistration.hs` - Registration type
- `src/System/Agents/OneShot.hs` - `executeToolCall`, `llmCallTool`
- `src/System/Agents/MCP/Server.hs` - `executeToolCall`

## Design Decisions

**Tradeoffs:**

1. **Replace vs Extend rtVal**: Replacing `rtVal` with `ToolExecutionContext` is cleaner because:
   - `rtVal` was only ever `ConversationId` in practice
   - `ToolExecutionContext` is a strict superset of what `ConversationId` provided
   - Simplifies type signatures throughout

2. **Required vs Optional Context**: Making context required:
   - Simplifies tool implementations (no Maybe handling)
   - Always available if needed
   - Small overhead when not used

### 3. `src/System/Agents/OneShot.hs`
- Updated `llmCallTool` to construct a `ToolExecutionContext` using the new `mkToolExecutionContext` helper
- Added `mkToolExecutionContext` helper function that creates a minimal context with generated session, conversation, and turn IDs

### 4. `src/System/Agents/MCP/Server.hs`
- Updated `llmCallTool` to construct and pass `ToolExecutionContext` (same approach as OneShot.hs)

### 5. `src/System/Agents/AgentTree/OneShotTool.hs`
- Updated `runSubAgent` to take `ToolExecutionContext` instead of `ConversationId`

### 6. `src/System/Agents/LLMs/OpenAI.hs`

### Key Design Decisions
- The `ToolExecutionContext` is constructed at the call sites (`llmCallTool` in both `OneShot.hs` and `MCP.Server.hs`) with generated IDs when the full session context isn't available at the point of tool execution
- The context provides a structured way to pass session metadata to tools without exposing these details to the LLM
- IO tools can now access the full execution context, allowing them to use session IDs, conversation IDs, turn IDs, and optionally full session data if needed
