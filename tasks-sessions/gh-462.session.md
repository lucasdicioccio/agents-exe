## Summary

Integrate the signal-based trajectory metrics from the paper **"Signals: Trajectory Sampling and Triage for Agentic Interactions"** (Chen et al., 2026) into the session-print headers, displaying key interaction and execution signals alongside existing token usage statistics.

## Background: Signal Taxonomy from Paper

The paper proposes a lightweight, signal-based framework for triaging agentic interaction trajectories. Signals are descriptive markers of recurring behavioral patterns, organized into three categories:

### 1. Interaction Signals (Learning-Oriented)
Derived from user-assistant natural language discourse:

| Signal | Description | Detection Approach |
|--------|-------------|-------------------|
| **Misalignment** | Semantic/intent mismatch (rephrasing, corrections, clarifications) | Phrase-level cues, local similarity checks |
| **Stagnation** | Discourse without visible progress (near-duplicate responses, circular explanations) | Discourse heuristics, repetition detection |
| **Disengagement** | Withdrawal of cooperative intent (exit requests, negative stance, abandonment) | Explicit markers: "talk to a human", negative sentiment |
| **Satisfaction** | Successful convergence (gratitude, success confirmations, closing utterances) | Positive confirmation patterns |

### 2. Execution Signals (Learning-Oriented)
Derived from structured runtime events (tool calls, API responses):

| Signal | Description | Detection Approach |
|--------|-------------|-------------------|
| **Failure** | Action attempts not yielding usable outcomes (empty results, no-op actions, errors) | Tool outcome classification |
| **Loop** | Repetitive execution without progress (retries, oscillations, parameter drift) | Sequence analysis over invocation streams |

### 3. Environment Signals (Diagnosis-Oriented)
System-level boundary conditions (not for learning):

| Signal | Description |
|--------|-------------|
| **Exhaustion** | Context overflows, rate limits, API failures, resource boundaries |

```haskell
-- | Signal taxonomy for trajectory analysis
-- Based on: "Signals: Trajectory Sampling and Triage for Agentic Interactions"
data InteractionSignals = InteractionSignals
    { sigMisalignmentCount :: Int
    -- ^ Semantic/intent mismatches detected (rephrasing, corrections)
    , sigStagnationCount :: Int
    -- ^ Discourse without progress (near-duplicates, circular explanations)
    , sigDisengagementDetected :: Bool
    -- ^ User withdrawal markers detected (exit requests, negative stance)
    , sigSatisfactionDetected :: Bool
    -- ^ Success confirmation markers detected (gratitude, confirmations)
    }
    deriving (Show, Eq, Generic)

data ExecutionSignals = ExecutionSignals
    { sigFailureCount :: Int
    -- ^ Tool/action attempts with non-advancing outcomes
    , sigLoopDetected :: Bool
    -- ^ Repetitive execution patterns without progress
    , sigLoopToolSequence :: [Text]
    -- ^ Tool names involved in detected loop (for diagnostics)
    }
    deriving (Show, Eq, Generic)

data EnvironmentSignals = EnvironmentSignals
    { sigExhaustionCount :: Int
    -- ^ Resource boundary hits (rate limits, context overflows)
    , sigExhaustionTypes :: [Text]
    -- ^ Types of exhaustion detected (e.g., "rate_limit", "context_overflow")
    }
    deriving (Show, Eq, Generic)

-- | Combined trajectory signals for a session
data TrajectorySignals = TrajectorySignals
    { trajInteraction :: InteractionSignals
    , trajExecution :: ExecutionSignals
    , trajEnvironment :: EnvironmentSignals
    -- | Composite informativeness score (0-100)
    -- Higher = more likely to contain actionable insights
    , trajInformativenessScore :: Int
    }
    deriving (Show, Eq, Generic)

-- | Per-step signal tracking (stored in StepByteUsage)
data StepSignals = StepSignals
    { stepInteractionSignals :: Maybe InteractionSignals
    , stepExecutionSignals :: Maybe ExecutionSignals
    , stepEnvironmentSignals :: Maybe EnvironmentSignals
    }
    deriving (Show, Eq, Generic)
```

Implement lightweight, rule-based signal detection (no LLM calls required):

```haskell
module System.Agents.Session.Signals (
    -- * Detection functions
    detectInteractionSignals,
    detectExecutionSignals,
    detectEnvironmentSignals,
    calculateTrajectorySignals,
    
    -- * Heuristics
    misalignmentHeuristics,
    stagnationHeuristics,
    loopDetectionHeuristics,
) where

-- Interaction signal detection from discourse
-- Uses phrase-level matching and local similarity checks
detectInteractionSignals :: [Turn] -> InteractionSignals

-- Execution signal detection from tool call patterns
detectExecutionSignals :: [Turn] -> ExecutionSignals

-- Environment signal detection from error patterns
detectEnvironmentSignals :: [Turn] -> EnvironmentSignals

-- Loop detection: repeated tool calls with identical or systematically varying inputs
-- Pattern rules:
--   - Repeated calls with identical inputs (exact loop)
--   - Repeated calls with systematically varying inputs (parameter drift)
--   - Repeated multi-tool cycles (oscillation)
detectExecutionLoop :: [LlmToolCall] -> Maybe ([Text], Int)  -- (tool names, loop count)
```

**Detection Heuristics (from paper):**

| Signal | Detection Rule | Example Patterns |
|--------|---------------|------------------|
| Misalignment | Phrase matching + similarity checks | "rephrase", "correction", "clarify", "I meant", "not what I asked" |
| Stagnation | Near-duplicate detection + turn count | Same assistant response within 3 turns, >5 turns without user progress |
| Disengagement | Terminal markers | "talk to human", "stop", "cancel", "this isn't working", explicit exit requests |
| Satisfaction | Success markers | "thank you", "that worked", "perfect", "done", "solved" |
| Failure | Tool outcome analysis | Empty results, HTTP 4xx/5xx, no-op responses, error messages |
| Loop | Sequence analysis | Same tool called >2x with identical/similar args, tool A→B→A pattern |
| Exhaustion | Error pattern matching | "context limit", "rate limit", "quota exceeded", "timeout" |

## Design Decisions

### 1. Signal Computation Timing

**Option A: Compute on-demand (Recommended)**
- Signals calculated when `session-print` or TUI renders
- No storage overhead
- Always reflects current heuristics
- Slightly higher render cost

**Option B: Store in session JSON**
- Add `sessionSignals` field to `Session` type
- Incrementally updated during session lifecycle
- Storage overhead per session
- Historical signal data preserved

**Recommendation:** Option A for initial implementation. Signals are lightweight to compute (no LLM calls), and on-demand computation ensures consistency with evolving heuristics.

### 2. Informativeness Score Calculation

Simple weighted aggregation from paper's approach:

```haskell
calculateInformativenessScore :: TrajectorySignals -> Int
calculateInformativenessScore signals =
    let is = trajInteraction signals
        es = trajExecution signals
        -- Positive signals (indicate actionable insights)
        failurePoints = min 30 (sigFailureCount es * 10)  -- Cap at 30
        misalignPoints = min 20 (sigMisalignmentCount is * 5)  -- Cap at 20
        loopPoints = if sigLoopDetected es then 20 else 0
        disengagePoints = if sigDisengagementDetected is then 15 else 0
        -- Satisfaction reduces informativeness (success case)
        satisfactionPenalty = if sigSatisfactionDetected is then -10 else 0
     in max 0 $ min 100 $ 30 + failurePoints + misalignPoints + loopPoints + disengagePoints + satisfactionPenalty
```

### 3. Backward Compatibility

- `StepSignals` stored as `Maybe` in `StepByteUsage` (similar to `stepTokenUsage`)
- `statTrajectorySignals` as `Maybe` in `SessionStatistics`
- Missing signal fields render as "Signals: not computed" or omitted
