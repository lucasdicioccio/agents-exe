### User Query

# [Design] Support recursive references in AgentTree

## Problem Statement

The current `AgentTree` implementation requires a fully-explored tree before execution. This prevents:

1. **Self-references**: An agent cannot call itself (direct recursion)
2. **Mutual recursion**: Agent A cannot call Agent B if B also calls A
3. **Circular tool dependencies**: Any cycle in the agent hierarchy is impossible

## Current Architecture Limitation

In `System.Agents.AgentTree`:

```haskell
-- loadAgentTree builds children BEFORE parent
loadAgentTree props tree = do
    agentRuntimes <- traverse (loadAgentTree props) tree.agentConfigChildren  -- children first
    -- ... then parent is created with references to children's runtimes
    rt <- initAgentTreeAgent ... okRuntimes ...  -- needs children runtimes
```

The `agentToTool :: Runtime -> AgentSlug -> AgentId -> ToolRegistration` function requires a fully-initialized `Runtime`, which needs all sub-agents upfront.

## Proposed Solutions

### Option 1: IORef/MVar-based Lazy Wiring (Recommended)

Use mutable references to break the cycle:

```haskell
data AgentTree = AgentTree
    { agentFile :: FilePath
    , agentBase :: Agent
    , agentRuntime :: Runtime.Runtime
    , agentChildren :: [AgentTree]
    , agentSelfReference :: IORef (Maybe Runtime.Runtime)  -- NEW
    }
```

**Two-phase initialization:**
1. Phase 1: Create all `Runtime` shells with empty tool lists
2. Phase 2: Wire up references using `IORef`/`MVar`

**Pros:** Minimal API changes, maintains current structure  
**Cons:** Requires careful handling of initialization order

### Option 2: Deferred Tool Resolution

Instead of passing `[Runtime]` to `initAgentTreeAgent`, pass a registry lookup:

```haskell
type AgentRegistry = AgentSlug -> IO (Maybe Runtime)

initAgentTreeAgent :: ... -> AgentRegistry -> ...
```

Tools would resolve agent references at call-time rather than init-time.

**Pros:** Clean separation, supports dynamic agent addition  
**Cons:** Requires refactoring tool registration API

### Option 3: Explicit Recursion Bound

Add configuration to specify recursion depth limits:

```haskell
data Agent = Agent
    { ...
    , maxRecursionDepth :: Maybe Int  -- NEW
    , allowSelfRecursion :: Bool      -- NEW
    }
```

Prevents infinite loops when cycles exist.

## Tradeoffs

| Approach | Complexity | API Change | Risk |
|----------|-----------|------------|------|
| IORef Lazy | Medium | Minimal | Initialization bugs |
| Deferred Resolution | Medium | Significant | Runtime failures |
| Recursion Bounds | Low | Minimal | False positives |

## Recommended Approach

Implement **Option 1 (IORef/MVar)** with **Option 3 (Recursion Bounds)** for safety:

1. Add `IORef (Map AgentSlug Runtime)` to `Props` for deferred lookup
2. Two-phase init: create shells → wire references
3. Add recursion tracking in `ToolExecutionContext`
4. Enforce depth limits at call time
