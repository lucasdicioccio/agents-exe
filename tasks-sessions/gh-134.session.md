## Summary

Add support for PostgREST as a new toolbox type, similar to the existing OpenAPI toolbox. PostgREST exposes PostgreSQL databases as REST APIs and provides a Swagger/OpenAPI spec at the root endpoint, but requires special handling for row filters, pagination, ordering, and column selection.

## Background

PostgREST (https://postgrest.org) is a standalone web server that turns PostgreSQL databases directly into RESTful APIs. Unlike generic OpenAPI:
- Endpoints map directly to database tables (e.g., `/users`, `/orders`)
- GET operations support special query parameters: `limit`, `offset`, `select`, `order`
- Column-based filtering is exposed via `rowFilter.` parameter references in the spec
- The OpenAPI description endpoint itself should be excluded from tool generation

    { tpFilters :: Maybe FilterSchema      -- ^ Column filters
    , tpSubset :: Maybe SubsetSchema       -- ^ limit, offset, columns
    , tpRanking :: Maybe RankingSchema     -- ^ order

data FilterSchema = FilterSchema
    { fsDescription :: Text
    , fsProperties :: Map Text ColumnFilterSchema
    }

    , ssColumns :: Maybe Text  -- ^ Description: "comma-separated column names"

### Key Differences from OpenAPI Toolbox

| Aspect | OpenAPI | PostgREST |
|--------|---------|-----------|
| Tool naming | `openapi_{toolbox}_{operationId}` | `postgrest_{toolbox}_{table}` |
| GET parameters | Individual params | Structured: filters/subset/ranking |
| Row filtering | Generic query params | Column-based with operators |
| Pagination | Custom implementation | Standard limit/offset |
| Ordering | Custom | Standard order parameter |
| Column selection | N/A | select parameter |

### Tool Parameter Structure

Each PostgREST GET tool should have these parameter groups:

```haskell
-- Example: get_users tool parameters
{
  "type": "object",
  "properties": {
    "filters": {
      "type": "object",
      "description": "Row filters on columns",
      "properties": {
        "name": {"type": "string", "description": "Filter on name column (string)"},
        "age": {"type": "string", "description": "Filter on age column (integer)"}
      }
    },
    "subset": {
      "type": "object", 
      "description": "Pagination and column subsetting",
      "properties": {
        "offset": {"type": "string", "description": "Number of rows to skip"},
        "limit": {"type": "string", "description": "Max rows to return"},
        "columns": {"type": "string", "description": "Comma-separated column names"}
      }
    },
    "ranking": {
      "type": "object",
      "description": "Ordering clause", 
      "properties": {
        "order": {"type": "string", "description": "e.g., created_at.desc,name.asc"}
      }
    }
  }
}
```

### Name Normalization

Use the same normalization as OpenAPI (`normalizeForLLM` from `System.Agents.Tools.OpenAPI.Converter`):
- Table paths like `/users` → `postgrest_mydb_get_users`
- Handle dots in schema-qualified tables: `/public.users` → `postgrest_mydb_get_public_users`

### Implementation Steps

1. **Types Module** (`Types.hs`):
   - Define `Config`, `RowFilter`, `Trace`, `PostgRESTool`
   - Define parameter schemas: `FilterSchema`, `SubsetSchema`, `RankingSchema`

2. **Converter Module** (`Converter.hs`):
   - Parse PostgREST OpenAPI spec

3. **Toolbox Module** (`PostgRESToolbox.hs`):
   - Fetch and parse spec
   - Initialize HTTP runtime
   - Create tool handlers
   - Build query strings from structured arguments
   - Execute requests and return results

4. **Integration**:
   - Export from `System.Agents.Tools` module
   - Add to tool registration system similar to `registerOpenAPITools`

### Tradeoffs Considered

1. **Simplified filter syntax**: Start with direct value passing (PostgREST infers `eq` operator). Advanced operators (gt, lt, like) can be passed as strings in filter values.

2. **Single GET per table**: Only generate GET tools initially. POST/PUT/PATCH/DELETE can be added later as they require different parameter structures.

3. **No schema introspection**: Rely on the OpenAPI spec rather than direct PostgreSQL introspection. This keeps the implementation consistent with the OpenAPI toolbox.

4. **Structured vs flat parameters**: Use structured `filters`/`subset`/`ranking` objects instead of flat parameters to:
   - Group related functionality
   - Avoid parameter name collisions with column names
   - Make the API more discoverable for LLMs

## Summary

I've successfully implemented PostgREST toolbox support for automatic database API mapping. This PR adds support for PostgREST as a new toolbox type, similar to the existing OpenAPI toolbox.

### New Files Created

1. **`src/System/Agents/Tools/PostgREST/Types.hs`** - PostgREST-specific types including:
   - `Config` - Configuration for connecting to PostgREST instances
   - `RowFilter` - Column-based row filter definitions
   - `Trace` - Trace events for monitoring operations
   - `PostgRESTool` - Tool representation for database tables
   - `ToolParameters`, `FilterSchema`, `SubsetSchema`, `RankingSchema` - Structured parameter schemas

2. **`src/System/Agents/Tools/PostgREST/Converter.hs`** - Conversion logic that:
   - Parses PostgREST OpenAPI specs
   - Converts table endpoints to tools (GET operations)
   - Extracts row filters from `rowFilter.` parameter references
   - Builds structured tool parameters (filters/subset/ranking)
   - Generates LLM-compatible tool names

3. **`src/System/Agents/Tools/PostgRESToolbox.hs`** - Runtime and execution module that:
   - Fetches PostgREST OpenAPI specifications
   - Converts table operations to executable tools
   - Handles HTTP execution with PostgREST-specific query parameter building
   - Supports structured query parameters for filters, subsetting, and ordering
   - Integrates with the LLM tool registration system

### Key Features

- **Structured Query Parameters**: Tools use three parameter groups:
  - `filters` - Column-based row filters (e.g., `name=eq.John`, `age=gt.18`)
  - `subset` - Pagination (limit/offset) and column selection (select)
  - `ranking` - Ordering clause (order parameter)

- **Tool Naming**: Tools are named `postgrest_{toolbox}_{method}_{table}` (e.g., `postgrest_localhost_get_users`)

- **JWT Authentication**: Supports Bearer token authentication via `configToken`

- **Row Filter Detection**: Automatically detects column filters from the OpenAPI spec

- **Path Filtering**: Skips non-table paths (root `/`, OpenAPI endpoints, Swagger UI)
