# Integrate BuiltinToolbox into Runtime initialization

Depends-on: #228
Base-branch: gh-builtin-toolbox
Final: false

## Task

Update `System.Agents.Runtime.Runtime` to initialize builtin toolboxes (specifically SQLite toolboxes) when creating a runtime.

## Background

The Runtime module is responsible for initializing all toolboxes and making them available to agents. We need to extend it to handle builtin toolboxes.

## Changes Required

### 1. Add SqliteToolbox import

```haskell
import qualified System.Agents.Tools.SqliteToolbox as SqliteToolbox
```

### 2. Add trace events for builtin toolboxes

Add to `System.Agents.Runtime.Trace`:

```haskell
data Trace
    = ...existing traces...
    | BuiltinToolboxTrace !Text !BuiltinToolbox.Trace  -- toolbox name, trace event
    | BuiltinToolboxInitError !Text !String  -- toolbox name, error message
    deriving (Show)
```

(Note: This may need to be a separate commit or the Trace module may need reorganization to avoid circular imports.)

### 3. Update newRuntime signature

```haskell
newRuntime ::
    AgentSlug ->
    AgentAnnounce ->
    Tracer IO Trace ->
    OpenAI.ApiKey ->
    OpenAI.Model ->
    ToolboxDirectory ->
    [IOToolBuilder] ->
    [McpToolConfig] ->
    [ToolRegistration] ->  -- OpenAPI tool registrations
    [SqliteToolbox.Toolbox] ->  -- NEW: SQLite toolboxes
    IO (Either String Runtime)
```

### 4. Update newRuntime implementation

In the `readAdditionalTools` function, add SQLite tool registration:

```haskell
readAdditionalTools :: IO [ToolRegistration]
readAdditionalTools = do
    mcpTools <- readMcpToolsRegistrations
    sqliteTools <- readSqliteToolsRegistrations
    -- OpenAPI tools are pre-registered and static
    pure $ mcpTools <> sqliteTools <> openApiToolRegs

readSqliteToolsRegistrations :: IO [ToolRegistration]
readSqliteToolsRegistrations = do
    results <- traverse registerSqliteTools sqliteToolboxes
    -- Collect successes, log failures
    let (errors, registrations) = partitionEithers results
    -- Trace errors if any
    forM_ errors $ \err -> 
        runTracer tracer (BuiltinToolboxInitError "sqlite" err)
    pure $ concat registrations
```

### 5. Handle SQLite toolbox initialization

The SQLite toolboxes should be initialized before calling `newRuntime`. The initialization should happen in the CLI layer (see #230), and the initialized toolboxes passed to `newRuntime`.

Alternatively, we could initialize them inside `newRuntime` by adding the descriptions to the signature:

```haskell
newRuntime ::
    ...
    [BuiltinToolboxDescription] ->  -- NEW: descriptions to initialize
    IO (Either String Runtime)
```

Then initialize inside:

```haskell
-- Initialize SQLite toolboxes
sqliteResults <- traverse (initializeSqliteToolbox tracer) sqliteDescriptions
let (sqliteErrors, sqliteToolboxes) = partitionEithers sqliteResults
mapM_ (\err -> runTracer tracer (BuiltinToolboxInitError "sqlite" err)) sqliteErrors
```

## Recommendation

Use the second approach (initialize inside newRuntime) for consistency with how other toolboxes are handled. This requires:

1. Adding `BuiltinToolboxDescription` import
2. Pattern matching on descriptions to initialize appropriate toolboxes
3. Error collection and tracing

## Files to Modify

- `src/System/Agents/Runtime/Runtime.hs`
- `src/System/Agents/Runtime/Trace.hs` (add new trace events)

## Testing

- Test runtime initialization with SQLite toolboxes
- Verify error handling when SQLite database doesn't exist
- Verify correct tool registration
- Test read-only vs read-write access modes

## Tradeoffs

- Initializing inside newRuntime keeps all toolbox initialization in one place
- Passing pre-initialized toolboxes gives more flexibility but requires more coordination in CLI layer
- Chose inside initialization for consistency with existing patterns
