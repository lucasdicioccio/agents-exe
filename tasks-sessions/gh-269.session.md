# Add LuaToolbox with inter-toolbox portal for agent-scriptable tool orchestration

---
## Summary

Add a **LuaToolbox** that provides a sandboxed Lua interpreter where LLM-generated scripts can orchestrate other tools. The key innovation is an **inter-toolbox portal** mechanism that allows Lua scripts to call other tools (PostgREST, Bash, IO, SQLite, etc.) through a standardized interface.

This enables agents to write complex, multi-step workflows (setup → probe → observe → act) in Lua rather than requiring specialized hardcoded tools.

## Motivation

Current toolboxes are isolated. The agent can use PostgREST toolbox OR Bash toolbox, but cannot easily combine them in a scripted workflow. For complex bug reproduction:

1. Set up environment (bash: database, services)
2. Configure agent dynamically (create JSON with discovered endpoints)
3. Run probes (HTTP calls via PostgREST toolbox)
4. Analyze results (Lua logic)
5. Patch code (file edits)
6. Loop until fixed

**Example: PostgREST login bug**
```lua
-- Agent-generated Lua script
local tools = require("tools")

-- Setup
tools.bash.run("createdb testdb && psql testdb < schema.sql")
tools.bash.run("postgrest postgrest.conf &")
tools.time.sleep(2)

-- Test with special characters
local result = tools.http.post("http://localhost:3001/rpc/login", {
  json = {email = "test@example.com", password = "special!@#chars"}
})

if result.status == 500 then
  -- Analyze logs and patch
  local logs = tools.fs.read("/var/log/postgrest.log")
  if string.find(logs, "syntax error") then
    tools.fs.patch("src/Auth.hs", "sanitize = id", "sanitize = T.escapeString")
    tools.bash.run("cabal build")
    return {fixed = false, retry = true}
  end
else
  return {fixed = true}
end
```

### 1. Tool Portal Mechanism

**Modified ToolExecutionContext:**
```haskell
data ToolExecutionContext = ToolExecutionContext
  { ctxSessionId :: SessionId
  , ctxConversationId :: ConversationId
  , ctxTurnId :: TurnId
  , ctxAgentId :: Maybe AgentId
  , ctxFullSession :: Maybe Aeson.Value
  , ctxToolPortal :: Maybe ToolPortal  -- NEW

-- Portal is a synchronous call: Lua waits for result
type ToolPortal = ToolCall -> IO ToolResult

data ToolCall = ToolCall
  { callToolName :: Text
  , callArgs :: Aeson.Value
```

### 4. Security Model: Whitelist Approach

**Upfront declaration:** Agent configuration declares which tools Lua can access:
```json
{
  "builtinToolboxes": [
    {
      "tag": "LuaToolbox",
      "contents": {
        "name": "lua",
        "description": "Sandboxed Lua interpreter",
        "maxMemoryMB": 256,
        "maxExecutionTimeSeconds": 300,
        "allowedTools": ["bash", "postgrest", "sqlite", "io"],
        "allowedPaths": ["./repro-cases", "./logs"],
        "allowedHosts": ["localhost", "127.0.0.1"]
      }
    }
  ]
}
```

**Lua sandbox restrictions:**
- Only `tools` module and standard library available
- No `os.execute`, `io.popen`, etc. (use `tools.bash` instead)
- No raw socket access (use `tools.http` or whitelisted tool)
- Memory limit enforced by Lua allocator hooks
- Timeout enforced by Haskell watchdog thread
- Filesystem restricted to `allowedPaths`
- Network restricted to `allowedHosts` (for direct http)

### 5. Tool Interface: Dynamic Dispatch

Each tool exports a handler for the portal:
```haskell
class ToolPortalHandler a where
  toolName :: a -> Text
  toolSchema :: a -> ToolSchema
  toolInvoke :: a -> Aeson.Value -> IO (Either ToolError Aeson.Value)

-- Registration at toolbox initialization
registerWithPortal :: ToolPortal -> [ToolPortalHandler] -> IO ()
```

**Dynamic dispatch by name:**
```haskell
portal :: [RegisteredTool] -> ToolCall -> IO ToolResult
portal tools call = do
  case find (\t -> toolName t == call.callToolName) tools of
    Just tool -> do
      start <- getCurrentTime
      result <- toolInvoke tool call.callArgs
      end <- getCurrentTime
      recordTrace call call.callerId result
      return $ ToolResult result (diffUTCTime end start) traceId
    Nothing -> 
      throw $ ToolNotAllowed call.callToolName
```

```lua
local tools = require("tools")
local json = require("json")
local fs = require("fs")
local time = require("time")

-- Configuration
local REPRO_DIR = "repro-2024-01-15.001"
local MAX_ITERATIONS = 10

-- Setup repro directory
fs.mkdir(REPRO_DIR)

-- Setup environment (with fallback to user prompt if needed)
local setup_ok = pcall(function()
  tools.bash.run("createdb qa_testdb 2>/dev/null || echo 'db exists'")
  tools.bash.run("psql qa_testdb < schema.sql")
  tools.bash.run("postgrest postgrest.conf &")
  time.sleep(2)
end)

if not setup_ok then
  return {
    error = "Setup failed. Please ensure PostgreSQL and PostgREST are installed.",
    needsUserIntervention = true
  }
end

-- Create repro script
fs.write(REPRO_DIR .. "/test_login.sh", [[
#!/bin/bash
curl -s -w "\nHTTP_CODE:%{http_code}" \
  -X POST http://localhost:3001/rpc/login \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "password": "special!@#chars"}'
]])

-- Test loop
for i = 1, MAX_ITERATIONS do
  local result = tools.bash.run("bash " .. REPRO_DIR .. "/test_login.sh")
  local http_code = string.match(result.stdout, "HTTP_CODE:(%d+)")
  local body = string.gsub(result.stdout, "\nHTTP_CODE:%d+", "")
  
  -- Log this attempt
  fs.write(REPRO_DIR .. "/logs/attempt-" .. i .. ".log", result.stdout)
  
  if http_code == "500" then
    -- Analyze error
    local logs = tools.bash.run("tail -50 /var/log/postgrest.log").stdout
    
    if string.find(logs, "syntax error at or near") then
      -- Fix SQL injection/escaping issue
      local patched, err = fs.patch(
        "src/Auth.hs",
        "sanitizePassword = id",
        "sanitizePassword = T.replace \"'\" \"''\""
      )
      
      if not patched then
        return {error = "Failed to patch: " .. err, iteration = i}
      end
      
      -- Rebuild
      local build = tools.bash.run("cabal build")
      if build.exitCode ~= 0 then
        return {error = "Build failed: " .. build.stderr, iteration = i}
      end
      
      -- Continue to next iteration
    else
      return {error = "Unknown error in logs", logs = logs, iteration = i}
    end
  elseif http_code == "200" then
    -- Success!
    local response = json.decode(body)
    if response.token then
      return {
        fixed = true,
        iterations = i,
        message = "Login works with special characters",
        reproDir = REPRO_DIR
      }
    end
  else
    return {error = "Unexpected HTTP code: " .. http_code, iteration = i}
  end
end

return {error = "Max iterations reached without fix", iterations = MAX_ITERATIONS}
```

## Comparison with Alternatives

### Why not Bash?

| Aspect | Bash | Lua |
|--------|------|-----|
| Data structures | Weak (text only) | Strong (tables) |
| Error handling | Messy (`set -e`, `$?`) | Clean (pcall) |
| JSON | Requires jq | Native module |
| Sandboxing | Very hard | Easier |
| Debugging | Hard | Stack traces |
| Return values | Exit codes + stdout | Rich data structures |

Bash is fine for simple sequences. Lua enables complex logic with data processing.

### Why not JavaScript/QuickJS?

- **Pros**: More familiar, better ecosystem, TypeScript possible
- **Cons**: Heavier dependency, more complex engine, JavaScript quirks
- **Verdict**: Lua is lighter and sufficient. JavaScript can be added later as alternative.

### Why not Python?

- **Pros**: Familiar, huge ecosystem
- **Cons**: Heavy dependency, complex sandboxing, slow startup
- **Verdict**: Too heavy for this use case.

## Future Extensions

- **PythonToolbox**: For ML/data science workflows
- **JavaScriptToolbox**: For web/Node.js workflows

All would use the same portal mechanism.
