# Implement Option C: Smart Detection with Explicit Binary Marking for Media Attachments

Depends-on: #265
Base-branch: gh-265-option-c
Final: false

## Overview

This issue implements **Option C** (Smart Detection with Explicit Binary Marking) from #265. After Phase 1 (safe UTF-8 decoding with lenient decode), we now need proper support for binary tool outputs and base64-encoded media attachments.

## Design Decisions

Based on project requirements:
1. **Extend `BlobToolSuccess`** with `Maybe MediaTypeHint` - GHC will guide refactoring
2. **Detection scope:** Common image types (PNG, JPEG, GIF, WebP) via magic bytes + Content-Type headers
3. **Include `MixedResponse`** from the start for multi-modal responses
4. **`UserToolResponse` must be LLM-agnostic** - conversion happens in `Session/OpenAI.hs` to allow moving conversations between LLM providers
5. **Backwards compatibility:** Use sum-type with custom JSON instances + add version number to `Session`

## Implementation Steps

```haskell
module System.Agents.Media.Types where

data MediaType
```

### Step 4: Update Conversion Functions (OneShot, MCP/Server)

Update `callResultToUserToolResponse` in both `OneShot.hs` and `MCP/Server.hs`:

```haskell
callResultToUserToolResponse :: OpenAI.ToolCall -> CallResult OpenAI.ToolCall -> UserToolResponse
callResultToUserToolResponse _ result =
    case result of
        BlobToolSuccess _ bytes mHint ->
            case mHint of
                Just (MediaTypeHint mediaType) ->
                    -- Binary media detected - encode as base64
                    MediaResponse $ MediaAttachment
                        { mediaMimeType = mediaTypeToMime mediaType
                        , mediaBase64Data = Text.decodeUtf8 (B64.encode bytes)
                        , mediaFilename = Nothing
                        }
                Just BinaryUnknown ->
                    -- Known binary but unknown type - could use generic application/octet-stream
                    -- or fall back to text with lenient decode
                    TextResponse $ Text.decodeUtf8With lenientDecode bytes
                Nothing ->
                    -- Try text first, fall back to media detection
                    case isValidUtf8 bytes of
                        True -> TextResponse $ Text.decodeUtf8 bytes
                        False -> 
                            -- Try magic byte detection
                            case detectFromMagicBytes bytes of
                                Just mediaType -> MediaResponse ...
                                Nothing -> TextResponse $ Text.decodeUtf8With lenientDecode bytes
        
        -- Structured results become JsonResponse
        OpenAPIToolResult _ toolResult -> JsonResponse $ Aeson.toJSON toolResult
        PostgRESToolResult _ toolResult -> JsonResponse $ Aeson.toJSON toolResult
        McpToolResult _ res -> JsonResponse $ Aeson.toJSON res
        SqliteToolResult _ toolResult -> JsonResponse $ Aeson.toJSON toolResult
        SystemToolResult _ toolResult -> JsonResponse $ Aeson.toJSON toolResult
        DeveloperToolResult _ valResult -> JsonResponse $ Aeson.toJSON valResult
        DeveloperToolScaffoldResult _ scaffoldResult -> JsonResponse $ Aeson.toJSON scaffoldResult
        LuaToolResult _ toolResult -> JsonResponse $ Aeson.toJSON toolResult
        
        -- Error cases become TextResponse
        ToolNotFound _ -> TextResponse "Tool not found"
        BashToolError _ err -> TextResponse $ Text.pack $ show err
        -- ... other errors
```

```haskell
-- | Convert UserToolResponse to OpenAI's tool response format.
-- This is where LLM-specific conversion happens.
toolResponseToMessages :: (LlmToolCall, UserToolResponse) -> [Aeson.Value]
toolResponseToMessages (LlmToolCall callVal, response) =
    let toolCallId = extractToolCallId callVal
        content = case response of
            TextResponse t -> Aeson.String t
            JsonResponse v -> Aeson.toJSON v
            MediaResponse media -> 
                -- OpenAI format for base64 images
                Aeson.object ["type" .= ("image_url" :: Text), 
                             "image_url" .= Aeson.object ["url" .= ("data:" <> media.mediaMimeType <> ";base64," <> media.mediaBase64Data)]]
            MixedResponse parts ->
                -- OpenAI's content array format
                Aeson.Array $ Vector.fromList $ map contentPartToOpenAI parts
    in [Aeson.object 
        [ "role" .= ("tool" :: Text)
        , "tool_call_id" .= toolCallId
        , "content" .= content
        ]]

contentPartToOpenAI :: ContentPart -> Aeson.Value
contentPartToOpenAI (TextPart t) = Aeson.object ["type" .= ("text" :: Text), "text" .= t]
contentPartToOpenAI (MediaPart m) = 
    Aeson.object ["type" .= ("image_url" :: Text), 
                 "image_url" .= Aeson.object ["url" .= ("data:" <> m.mediaMimeType <> ";base64," <> m.mediaBase64Data)]]
```

## Key Trade-offs Documented

1. **IO in detection:** We allow IO in `detectMediaType` for future `libmagic` integration, but implement pure magic byte detection first.

2. **LLM-agnostic design:** `UserToolResponse` is opaque to LLM specifics. If we add Claude/Anthropic support, we add conversion functions in their respective modules without changing Session types.

3. **Session versioning:** Adding `sessionVersion` allows future schema changes. Version 0 = legacy (before this change), Version 1 = with new `UserToolResponse`.

4. **MixedResponse complexity:** Included from start but tools can stick to single-type responses until multi-modal is needed.
