## State Management Considerations

The `Agent` type currently doesn't carry state like `AgentId` or `ConversationId`. We need to ensure these are available:

Options:
1. **Add to Agent record**: Clean but changes Agent definition
2. **Pass at call site**: Requires caller to have this info (current approach in OneShot)
3. **ReaderT pattern**: More complex but flexible

Recommended: Option 2 for minimal changes - the caller (OneShot, MCP Server) already has this information.

## Design Decisions

**Tradeoffs:**

1. **Always build full context vs lazy**: Build at call site because:
   - Context construction is cheap (mostly just copying references)
   - Always available when needed
   - No lazy IO complications

2. **Include full session by default**: No, make it opt-in:
   - Sessions can grow large
   - Most tools don't need full history
   - Can be enabled per-agent or per-runtime

3. **AgentId in Agent vs passed separately**: Passed separately because:
   - Agent type is currently pure configuration
   - AgentId is runtime state
   - Keeps concerns separated
