# Extend Session types to track parent-child relationships for nested agent calls

Depends-on: 
Base-branch: gh-347-recursive-conversations
Final: false

## Background

This issue is part of implementing proper LLM-to-LLM recursive conversation capture (#347).

Currently, when an agent calls another agent via `turnAgentRuntimeIntoIOTool`, the sub-agent creates its own session, but there is no link back to the parent session. This makes it impossible to:
- Display conversation hierarchies in the TUI
- Trace the full call chain for debugging
- Understand the relationship between parent and child conversations

## Task

Extend the Session types to track parent-child relationships for nested agent calls.

## Changes Required

### In `src/System/Agents/Session/Types.hs`

Add the following fields to the `Session` data type:

```haskell
data Session = Session
    { turns :: [Turn]
    , sessionId :: SessionId
    , forkedFromSessionId :: Maybe SessionId
    , turnId :: TurnId
    -- NEW FIELDS:
    , parentSessionId :: Maybe SessionId
    -- ^ The session ID of the parent session (if this is a sub-agent call)
    , parentConversationId :: Maybe ConversationId
    -- ^ The conversation ID of the parent (for tracking call hierarchy)
    , parentAgentSlug :: Maybe Text
    -- ^ The slug of the agent that initiated this sub-agent call
    }
```

**Important**: Update the `ToJSON` and `FromJSON` instances to handle backward compatibility with existing sessions that don't have these fields.

### Add helper functions in `src/System/Agents/Session/Base.hs`

```haskell
-- | Create a child session linked to a parent session
mkChildSession :: 
    SessionId      -- ^ Parent session ID
    -> ConversationId -- ^ Parent conversation ID  
    -> Text         -- ^ Parent agent slug
    -> IO Session
mkChildSession parentSessId parentConvId parentSlug = do
    sessId <- newSessionId
    tId <- newTurnId
    pure $ Session
        { turns = []
        , sessionId = sessId
        , forkedFromSessionId = Nothing
        , turnId = tId
        , parentSessionId = Just parentSessId
        , parentConversationId = Just parentConvId
        , parentAgentSlug = Just parentSlug
        }

-- | Check if a session is a child session (was created by a parent agent)
isChildSession :: Session -> Bool
isChildSession = isJust . parentSessionId

-- | Get the root conversation ID for a session (follows parent chain)
getRootConversationId :: SessionStore -> Session -> IO ConversationId
getRootConversationId store sess = 
    case parentConversationId sess of
        Nothing -> pure $ ctxConversationId sess  -- session's own conv ID
        Just parentConvId -> do
            -- Could recursively look up parent session
            pure parentConvId
```

## Acceptance Criteria

- [ ] `Session` type includes `parentSessionId`, `parentConversationId`, and `parentAgentSlug` fields
- [ ] JSON serialization/deserialization handles backward compatibility
- [ ] Helper functions `mkChildSession`, `isChildSession` are implemented
- [ ] Existing tests pass with the new fields
- [ ] New fields are properly exported from `System.Agents.Session.Base`

## Tradeoffs

1. **Backward Compatibility**: The `FromJSON` instance must handle existing session files that don't have the new fields. Use `.:?` (optional field parsing) with `Maybe` defaults.

2. **Storage Overhead**: Each session now stores 3 additional Maybe fields. This is minimal overhead for the benefit of call chain tracking.

3. **Circular References**: The parent-child relationship creates a logical tree, but we don't enforce this at the type level. The caller is responsible for not creating circular references.
