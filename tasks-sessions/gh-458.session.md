## Summary

Replace the current manual file path text input dialog (`renderFilePathDialog`) in `src/System/Agents/TUI/Render.hs` with the proper `Brick.Widgets.FileBrowser` widget. This will provide a rich file browsing experience with directory navigation, file search, and visual file selection.

## Current State

The current implementation (`renderFilePathDialog`) provides a basic text editor where users manually type file paths:

```haskell
-- In src/System/Agents/TUI/Render.hs
renderFilePathDialog :: TuiState -> Widget N
renderFilePathDialog st =
    center $
        withAttr dialogAttr $
            borderWithLabel (txt " Attach File (Ctrl+F) ") $
                vBox
                    [ txt "Enter file path (or mime/type;path for explicit type):"
                    , txt ""
                    , hLimit 60 $
                        renderEditor
                            (txt . Text.unlines)
                            True
                            (st ^. tuiUI . filePathInput)
                    , txt ""
                    , txt "Enter: confirm  Esc: cancel"
                    ]
```

**Problems with current approach:**
- Users must know and type exact file paths
- No visual directory navigation
- No file search/filtering capability
- No visual feedback for file types or existence
- Error-prone manual path entry

## Target State

Use `Brick.Widgets.FileBrowser` to provide:
- Visual directory tree navigation with cursor selection
- Built-in search functionality (press `/` to search)
- File type icons/colors based on file type
- Directory entries marked with `/` suffix
- Selected files marked with `*` suffix
- Current working directory header
- File info footer (size, type)
- Keyboard shortcuts: Enter to select, Space to toggle, `/` to search, Esc to cancel

-- NEW: Handle FileBrowser dialog events
handleFileBrowserDialogEvent :: BrickEvent N AppEvent -> EventM N TuiState ()
handleFileBrowserDialogEvent ev =
    case ev of
        -- Cancel with Esc
        VtyEvent (Vty.EvKey Vty.KEsc []) -> do
            closeFileBrowserDialog
            showStatus StatusInfo "Attachment cancelled"
        
        -- Confirm selection with Enter
        VtyEvent (Vty.EvKey Vty.KEnter []) -> do
            handleFileBrowserSelection
        
        -- Handle FileBrowser-specific events
        VtyEvent vtyEv -> do
            mFb <- use (tuiUI . fileBrowser)
            case mFb of
                Nothing -> pure ()  -- Should not happen
                Just fb -> do
                    -- Handle file browser events (navigation, search, etc.)
                    newFb <- liftIO $ handleFileBrowserEvent vtyEv fb
                    tuiUI . fileBrowser .= Just newFb
        
        _ -> pure ()

-- NEW: Handle file selection from FileBrowser
handleFileBrowserSelection :: EventM N TuiState ()
handleFileBrowserSelection = do
    mFb <- use (tuiUI . fileBrowser)
    case mFb of
        Nothing -> do
            closeFileBrowserDialog
            showStatus StatusError "File browser not initialized"
        Just fb -> do
            -- Get selected file(s) - use fileBrowserSelection for multi-select
            -- or fileBrowserCursor for single cursor position
            case fileBrowserCursor fb of
                Nothing -> do
                    closeFileBrowserDialog
                    showStatus StatusWarning "No file selected"
                Just fileInfo -> do
                    let filePath = fileInfoFilePath fileInfo
                    -- Load the file as MediaAttachment
                    result <- liftIO $ loadFileAsAttachment filePath
                    case result of
                        Left err -> do
                            closeFileBrowserDialog
                            showStatus StatusError $ Text.pack err
                        Right attachment -> do
                            mConv <- getFocusedConversation
                            case mConv of
                                Nothing -> do
                                    closeFileBrowserDialog
                                    showStatus StatusError "No conversation selected"
                                Just conv -> do
                                    let convId = conversationId conv
                                    tuiUI . attachedFiles %= Map.insertWith (\new old -> old ++ new) convId [attachment]
                                    closeFileBrowserDialog
                                    showStatus StatusInfo $ "Attached: " <> maybe "unnamed" id attachment.mediaFilename

-- NEW: Load file as MediaAttachment
loadFileAsAttachment :: FilePath -> IO (Either String MediaAttachment)
loadFileAsAttachment filePath = do
    -- Detect MIME type from extension
    let mimeType = detectMimeType filePath
    -- Read file content
    fileContent <- ByteString.readFile filePath
    let base64Data = TextEncoding.decodeUtf8 $ Base64.encode fileContent
    let filename = Just $ Text.pack $ takeFileName filePath
    pure $ Right $ MediaAttachment mimeType base64Data filename

-- NEW: Simple MIME type detection from file extension
detectMimeType :: FilePath -> Text
detectMimeType fp = 
    let lowerFp = map toLower fp
    in case takeExtension lowerFp of
        ".png" -> "image/png"
        ".jpg" -> "image/jpeg"
        ".jpeg" -> "image/jpeg"
        ".gif" -> "image/gif"
        ".webp" -> "image/webp"
        ".svg" -> "image/svg+xml"
        ".mp3" -> "audio/mp3"
        ".mp4" -> "video/mp4"
        ".pdf" -> "application/pdf"
        ".json" -> "application/json"
        ".txt" -> "text/plain"
        ".md" -> "text/markdown"
        ".html" -> "text/html"
        _ -> "application/octet-stream"

## Key Design Decisions

### File Selection Strategy

Use `selectNonDirectories` as the selection predicate:
- Allows users to navigate into directories (Enter on directory changes into it)
- Only files are "selectable" with Enter/Space
- This is the standard behavior for file pickers

### MIME Type Detection

Implement simple extension-based MIME type detection:
- Covers common image types (PNG, JPEG, GIF, WebP, SVG)
- Covers common document types (PDF, JSON, TXT, MD)
- Falls back to `application/octet-stream` for unknown types
- Can be extended for audio/video types as needed

### Backward Compatibility

Keep `AttachmentDialogPathInput` state:
- Allows users to type explicit paths if needed
- Supports the "mime/type;path" syntax for explicit MIME type specification
- Can be toggled with a keybinding (e.g., Ctrl+T in dialog)

### Error Handling

The FileBrowser handles IOExceptions internally:
- Check `fileBrowserException` after directory changes
- Display error in status bar if exception occurs
- Always shows `..` entry for navigation even on errors

## Trade-offs and Considerations

### Pros
- Much better user experience with visual navigation
- Built-in search functionality
- Consistent with Brick's design patterns
- Rich visual feedback (file types, sizes, selection state)

### Cons
- Adds dependency on Brick's FileBrowser module (already available in brick-2.11)
- Requires IO for initialization (need `newFileBrowser`)
- Slightly more complex state management
- Default file filter may need tuning for different use cases

### Alternative: Keep Text Input

If full FileBrowser is too heavy, consider enhancing the text input with:
- Tab completion for file paths
- File existence validation
- Path history

However, the full FileBrowser is recommended for best UX.
