# Multi-location SessionStore: Support reading sessions from multiple directories with unified write prefix


## Overview

Enhance the `SessionStore` to support a unified view of sessions scattered across multiple locations in a project. This allows users to organize sessions logically while maintaining a single coherent view for searching, browsing, and resuming conversations.

## Motivation

Currently, sessions are stored in a single location determined by the `storePrefix`. Users often want to:

1. Store personal sessions in `~/.config/agents-exe/sessions/` (global)
2. Keep project-local sessions in `./sessions/` or `./.agents-sessions/` (git-ignored)
3. Have task-specific sessions in `./task-sessions/` for organizational purposes

The new `SessionStore` should unify all these locations for reading while writing to a single designated location.

## Proposed Configuration

Extend `agents-exe.cfg.json` with a new `sessions` configuration section:

```json
{
  "agentsConfigDir": "~/.config/agents-exe",
  "agentsFiles": [],
  "sessions": {
    "writeLocation": "./sessions/",
    "readLocations": [
      "./sessions/",
      "./.agents-sessions/",
      "./task-sessions/",
      "~/.config/agents-exe/sessions/"
    ]
  }
}
```

### Configuration Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `sessions.writeLocation` | `FilePath` | Yes | Directory where new sessions are written |
| `sessions.readLocations` | `[FilePath]` | Yes | List of directories to search for existing sessions |

### Backwards Compatibility

- If `sessions` section is missing, fall back to `agentsLogs.logSessionsJsonPrefix` (current behavior)
- If no prefix is configured at all, use `~/.config/agents-exe/sessions/` as default

### 1. SessionStore Type Refactoring

**File: `src/System/Agents/SessionStore.hs`**

```haskell
-- Current
newtype SessionStore = SessionStore
    { storePrefix :: FilePath
    }

-- Proposed
data SessionStore = SessionStore
    { sessionWritePrefix :: FilePath
    -- ^ Directory where new sessions are written
    , sessionReadPrefixes :: [FilePath]
    -- ^ Directories to search for existing sessions (in priority order)
    }

-- Smart constructor
mkSessionStore :: FilePath -> [FilePath] -> SessionStore
mkSessionStore writePrefix readPrefixes = SessionStore
    { sessionWritePrefix = writePrefix
    , sessionReadPrefixes = if writePrefix `elem` readPrefixes
                            then readPrefixes
                            else writePrefix : readPrefixes
    }

-- Simple constructor for single-location (backwards compat)
mkSimpleSessionStore :: FilePath -> SessionStore
mkSimpleSessionStore prefix = mkSessionStore prefix [prefix]
```

### 2. File Discovery Updates

The `findSessionFiles` function needs to aggregate sessions from all read locations:

```haskell
-- Key considerations:
-- 1. Sessions from different locations may have the same filename
-- 2. Duplicates should be resolved (last location wins or use full path as key)
-- 3. Each session maintains its original path for reading/writing updates

findSessionFiles :: SessionStore -> IO [SessionFileInfo]
findSessionFiles store = do
    -- Collect sessions from all read locations
    allSessions <- concat <$> mapM findSessionsInDir store.sessionReadPrefixes
    -- Deduplicate by ConversationId (keep first occurrence = highest priority)
    let deduplicated = dedupeBy (sessionInfoConversationId) allSessions
    -- Sort by modification time
    pure $ sortOn (Down . sessionInfoModTime) deduplicated
```

**Deduplication Strategy:**
- Use `ConversationId` (UUID) as the unique key
- First location in `readLocations` has highest priority
- If same session exists in multiple locations, the first one wins
- This allows "overriding" global sessions with local ones

### 3. Session Writing Behavior

```haskell
storeSession :: SessionStore -> ConversationId -> Session -> IO ()
storeSession store convId session = do
    -- Always write to the write prefix location
    let path = sessionWritePrefix store </> conversationIdToFileName convId
    storeSessionToFile sess path
```

### 4. Session Reading Behavior

```haskell
readSession :: SessionStore -> ConversationId -> IO (Maybe Session)
readSession store convId = do
    -- First check write prefix (most likely location)
    let writePath = sessionWritePrefix store </> conversationIdToFileName convId
    mSession <- readSessionFromFile writePath
    case mSession of
        Just session -> pure (Just session)
        Nothing -> 
            -- Search through read locations in order
            findFirstSession readLocations
```

## Trade-offs and Design Decisions

### Deduplication: First Location Wins

**Decision:** When the same session (by UUID) exists in multiple locations, the first location in `readLocations` takes precedence.

**Rationale:**
- Allows users to "override" global sessions with local versions
- Deterministic and predictable
- Matches common configuration file precedence patterns

### Write Location Not Automatically in Read Locations

**Decision:** If `writeLocation` is not explicitly listed in `readLocations`, it is automatically prepended.

**Rationale:**
- Most common use case: write and read from same location
- Users can still exclude write location if desired (by not listing it)

### No Automatic Directory Creation

**Decision:** The application will NOT automatically create session directories.

**Rationale:**
- Avoids cluttering filesystem with empty directories
- User explicitly controls directory structure
- Error messages guide users to create directories if needed
