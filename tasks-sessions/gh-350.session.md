# OS Model: Concurrent Access and Serialization

Depends-on: #2
Base-branch: gh-os-architecture
Final: false

## Overview

Design concurrent access patterns for shared resources. This builds on the Resource Management system to provide safe concurrent access to toolbox resources.

## Background

Multiple agents may need concurrent access to:
- Same SQLite database (readers can be concurrent, writers must be exclusive)
- HTTP connection pools (can be shared across concurrent requests)
- Lua interpreters (typically exclusive access per execution)
- Process handles (exclusive per process)

The current implementation uses MVars within each toolbox. The new system should use STM for composability and support multiple access patterns.

## Types to Define

### Access Patterns
```haskell
-- | Access pattern for a resource
data AccessPattern
  = ExclusiveAccess      -- Only one accessor (MVar-like)
  | ReadWriteAccess      -- Multiple readers, single writer
  | PoolAccess Int       -- Connection pool with max size
  | StatelessAccess      -- No synchronization needed

-- | Resource access control configuration
newtype AccessControl = AccessControl
  { accessPattern :: AccessPattern
  , accessTimeout :: Maybe NominalDiffTime
  } deriving (Show)

instance Component AccessControl where
  componentId _ = ComponentTypeId 20
```

### Synchronization Primitives
```haskell
-- STM-based read-write lock
data RWLock = RWLock
  { rwLockReaders :: TVar Int
  , rwLockWriter :: TVar Bool
  , rwLockWaitQueue :: TVar [TMVar ()]
  }

-- Resource synchronization state
newtype ResourceSync = ResourceSync
  { syncLock :: SyncPrimitive
  } deriving (Show)

instance Component ResourceSync where
  componentId _ = ComponentTypeId 21

data SyncPrimitive
  = ExclusiveLock (TMVar ())
  | ReadWriteLock RWLock
  | PoolLock (TBQueue ResourceToken)
  | NoLock
  deriving (Show)

data ResourceToken = ResourceToken
  { tokenResource :: ResourceId
  , tokenRelease :: STM ()
  }
```

### Resource Access Monad
```haskell
newtype ResourceM a = ResourceM
  { unResourceM :: ReaderT ResourceContext (ExceptT ResourceError IO) a }

data ResourceError
  = ResourceNotFound ResourceId
  | ResourceAccessTimeout ResourceId NominalDiffTime
  | ResourceScopeInvalid ResourceId [ScopeLevel]
  | ResourceClosed ResourceId
  | ResourceBusy ResourceId
  deriving (Show, Eq)

-- Operations
withExclusive :: ResourceId -> ResourceM a -> ResourceM a
withRead :: ResourceId -> ResourceM a -> ResourceM a
withWrite :: ResourceId -> ResourceM a -> ResourceM a
withPooled :: ResourceId -> ResourceM a -> ResourceM a
```

## Tradeoffs

1. **STM vs MVar**: STM chosen for composable transactions and automatic retry on conflict. MVar is slightly faster but less composable.

2. **Fairness**: Current RWLock design can starve writers. Consider writer-priority variant if contention becomes an issue.

3. **Pool Sizing**: Static pool sizes are simpler but may waste resources. Dynamic sizing could be added later.

4. **Timeout Handling**: Optional timeouts allow callers to decide between waiting and failing fast.
