### 💬 User Query

# SQLite Sandbox with Versioned Copies per Tool Call

Depends-on: 
Base-branch: gh-496
Final: false

## Overview
Implement automated snapshots after each SQLite write operation, providing full history and rollback capability for tool calls.

## Motivation
Currently, SQLite database modifications are permanent and immediate. This feature enables:
- **Auditability**: Full history of database changes
- **Rollback capability**: Revert to previous states
- **Experimentation**: Safe testing with ability to undo
- **Debugging**: Trace data modifications across tool calls

## Proposed Solution
Implement versioned copies of SQLite databases that are snapshotted after each write operation within a tool call context.

### Key Features
1. **Automatic Snapshots**: Create database snapshot after each write transaction
2. **Version History**: Maintain chain of database versions
3. **Rollback Support**: Ability to revert to any previous version
4. **Storage Efficiency**: Use copy-on-write or similar efficient storage mechanism

### Implementation Sketch
```haskell
-- New types for versioned SQLite
 data VersionedDB = VersionedDB {
     current :: Database,
     history :: [Snapshot],
     maxVersions :: Int  -- retention policy
 }

-- Snapshot after each write
snapshotVersion :: VersionedDB -> IO VersionedDB

-- Rollback to specific version
rollbackTo :: VersionId -> VersionedDB -> IO VersionedDB
```

## Acceptance Criteria
- [ ] SQLite writes automatically create versioned snapshots
- [ ] Can list available versions for a database
- [ ] Can rollback to any previous version
- [ ] Configurable retention policy (max number of versions)
- [ ] Storage overhead is reasonable (tested with typical workloads)
- [ ] Documentation updated with usage examples
