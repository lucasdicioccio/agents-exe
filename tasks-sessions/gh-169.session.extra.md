## Bug Fix: PostgREST Body Parameter Schema Resolution

### The Problem
PostgREST uses a two-level reference pattern for body parameters:
1. Operations reference `#/parameters/body.tablename`
2. The parameter definition contains `in: "body"` and the schema reference

The previous code only handled inline body parameters, not these reference-based patterns. This caused `opRequestBody` to be `Nothing` for PostgREST write operations (POST/PUT/PATCH), meaning LLMs couldn't see the request body schemas.

### Changes Made

1. **Extended `OpenAPISpec`** (`src/System/Agents/Tools/OpenAPI/Types.hs`):
   - Added `specParameters :: Maybe (Map Text Parameter)` field to store Swagger 2.0 parameters section
   - Updated parser to populate this field during Swagger 2.0 spec parsing
   - Added `ParamInBody` constructor to `ParamLocation` for body parameters

2. **Added resolution logic** (`src/System/Agents/Tools/PostgREST/Converter.hs`):
   - Added `isBodyParamRef` - checks if a parameter is a body parameter reference (`#/parameters/body.*`)
   - Added `resolveBodyParameterRef` - resolves the reference to get the actual schema from the spec's parameters section
   - Updated `convertPostgRESToTools` to use `extractRequestBodySchemaWithSpec` which tries standard extraction first, then falls back to PostgREST-specific resolution

3. **Added comprehensive tests** (`test/PostgRESTParseTest.hs`):
   - Test parsing PostgREST body parameter pattern
   - Test identifying body parameter references
   - Test resolving body parameter references to schemas
   - Test converting PostgREST specs with request body schemas

4. **Added test fixture** (`test/data/postgrest-body-test.json`):
   - Minimal PostgREST-style spec with the two-level body parameter pattern
   - Used for testing the fix without relying on the full monstergame.json

The fix ensures that POST/PUT/PATCH tools for PostgREST APIs now correctly expose request body schemas, allowing LLMs to know what fields to include when creating or updating records.
