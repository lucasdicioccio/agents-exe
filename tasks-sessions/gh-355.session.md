# OS Model: Persistence Layer for Sessions and Tool Calls

Depends-on: #5
Base-branch: gh-os-architecture
Final: false

## Overview

Implement a persistence layer for the OS model that can store sessions, tool calls, and entity states in an RDBMS. This enables durable state and querying across restarts.

## Background

Current `SessionStore` only persists session files. We need:
- Durable storage for all OS state (agents, toolboxes, conversations, turns, tool calls)
- Query capabilities for analytics and debugging
- Migration support from file-based to RDBMS storage

class Monad m => Persistence m where

### Persistable Components
```haskell
-- | Marker class for components that can be persisted
class (Component a, ToJSON a, FromJSON a) => Persistable a where
  persistenceTable :: Proxy a -> Text
  persistenceKey :: Proxy a -> EntityId -> Text

-- Instance for key components
instance Persistable AgentConfig where
  persistenceTable _ = "agents"
  persistenceKey _ eid = "agent:" <> entityIdToText eid

instance Persistable ConversationConfig where
  persistenceTable _ = "conversations"
  persistenceKey _ eid = "conv:" <> entityIdToText eid

instance Persistable Message where
  persistenceTable _ = "messages"
  persistenceKey _ eid = "msg:" <> entityIdToText eid

-- ... similar for other persistable components
```

data PersistenceEvent

## Database Schema (SQLite/Postgres)

```sql
-- Entities table (sparse representation)
CREATE TABLE entities (
  entity_id TEXT PRIMARY KEY,
  entity_type TEXT NOT NULL,  -- agent, toolbox, conversation, etc.
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Components table (JSON storage for flexibility)
CREATE TABLE components (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  entity_id TEXT NOT NULL REFERENCES entities(entity_id),
  component_type INTEGER NOT NULL,
  component_data JSON NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(entity_id, component_type)
);

-- Events table (append-only log)
CREATE TABLE events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  event_type TEXT NOT NULL,
  event_data JSON NOT NULL,
  entity_id TEXT REFERENCES entities(entity_id),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Messages table (optimized for conversation queries)
CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  conversation_id TEXT NOT NULL,
  turn_id TEXT NOT NULL,
  role TEXT NOT NULL,
  content TEXT NOT NULL,
  tool_calls JSON,
  timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Tool calls table (lineage tracking)
CREATE TABLE tool_calls (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  tool_call_id TEXT NOT NULL UNIQUE,
  turn_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  tool_input JSON NOT NULL,
  tool_result JSON,
  parent_call_id TEXT,
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  status TEXT NOT NULL
);

-- Indexes
CREATE INDEX idx_components_entity ON components(entity_id);
CREATE INDEX idx_components_type ON components(component_type);
CREATE INDEX idx_events_entity ON events(entity_id);
CREATE INDEX idx_messages_conversation ON messages(conversation_id);
CREATE INDEX idx_messages_turn ON messages(turn_id);
CREATE INDEX idx_tool_calls_turn ON tool_calls(turn_id);
CREATE INDEX idx_tool_calls_parent ON tool_calls(parent_call_id);
```

## Tradeoffs

1. **JSON vs Normalized**: JSON storage in components table provides flexibility but loses queryability. Consider partial normalization for frequently-queried fields.

2. **Sync vs Async**: Synchronous persistence is simpler but may slow down the OS. Consider async with durability guarantees.

3. **Single vs Multi-Backend**: Supporting multiple backends adds complexity. Start with SQLite, add Postgres later.
