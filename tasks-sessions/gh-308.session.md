## Summary

I implemented tool schema validation to help LLM self-correct when making incorrect tool calls. The feature includes:

### New Modules

1. **`System.Agents.Tools.Validation`** - Core validation module with:
   - `validateToolInput` - validates JSON against tool schemas
   - Support for all ParamType variants (String, Number, Boolean, Enum, Object, etc.)
   - Required vs optional property validation
   - Nested object validation
   - `formatValidationErrors` - generates helpful error messages for LLM correction

2. **`System.Agents.CLI.CheckToolCall`** - CLI command handler for:
   ```
   cat payload.json | agents-exe check-tool-call --tool <path-to-tool>
   ```

### Modified Files

- **`System.Agents.ToolRegistration`** - Exposed `mapArg` helper for schema conversion
- **`app/Main.hs`** - Added new `check-tool-call` command with `--tool` option
- **`agents.cabal`** - Added new modules to build

### Usage Example

```bash
# Validate a tool call payload
echo '{"name": "John", "age": 30}' | agents-exe check-tool-call --tool ./tools/my-tool

# Invalid payload returns helpful error message
echo '{"name": 123}' | agents-exe check-tool-call --tool ./tools/my-tool
# Output: Tool call validation failed with 1 error:
# 1. name: Expected string but got number
# Please correct these issues and try again.
```

### Key Features

- **Type validation**: Checks JSON values match expected types (string, number, boolean, enum, object)
- **Required fields**: Ensures all required properties are present
- **Extra properties**: Detects unknown properties not defined in schema
- **Nested objects**: Validates nested object structures recursively
- **LLM-friendly errors**: Clear, actionable error messages help LLMs self-correct
