## Summary

Add a `--agent <agent-slug>` command line argument that allows users to select which agent to use by its slug, rather than having to specify the full `--agent-file` path.

## Motivation

Currently, users must specify agent files explicitly with `--agent-file path/to/agent.json`. When running outside a project, users have agents in `~/.config/agents-exe/default/` but need to remember the full paths. Allowing selection by slug (e.g., `--agent openai-assistant`) would be much more user-friendly.

## Proposed Solution

Add a new command-line argument `--agent <slug>` that:
1. Looks up the agent by slug among all loaded agents
2. Selects that agent for the command (run, tui, check, etc.)
3. Fails with a helpful error if the slug is not found

## Usage Examples

```bash
# Run with a specific agent by slug
agents-exe run --agent openai-assistant -p "Hello world"

# Use in TUI mode
agents-exe tui --agent mistral-assistant

# Check a specific agent
agents-exe check --agent orchestrator

# Can still combine with --agent-file for additional agents
agents-exe run --agent-file ./custom.json --agent openai-assistant -p "Hello"
```

### Approach: Pre-load to resolve slugs

Since slugs are inside the agent files, we need to load them to resolve the slug. Two approaches:

**Option A: Filter after loading (recommended)**
1. Load all agents as usual
2. Filter to only use the matching slug
3. If no match, show error with available slugs

**Option B: Smart file selection**
1. Parse agent files to extract slugs without full runtime initialization
2. Select matching file paths
3. Continue with normal loading

Option A is simpler and reuses existing loading infrastructure.
