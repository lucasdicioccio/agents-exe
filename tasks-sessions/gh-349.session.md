## Overview

Define the resource management system with multiple lifecycle scopes and proper cleanup. This issue depends on the Core Entity types being defined first.

## Background

Tools require resources like:
- **SQLite**: file handles, prepared statements
- **Lua**: interpreter states
- **HTTP**: connection managers
- **Processes**: file descriptors

These resources have different lifecycles:
- **per-program**: Global HTTP connection pool
- **per-agent**: Agent-specific sandbox
- **per-toolbox**: SQLite connection per toolbox instance
- **per-conversation**: Isolated Lua state
- **per-turn**: Temporary resources
- **per-toolcall**: Single-use resources

## Types to Define

### Scope Definition
```haskell
-- | Defines the lifetime of a resource
newtype ResourceScope = ResourceScope
  { scopeHierarchy :: [ScopeLevel] }

data ScopeLevel
  = ProgramScope
  | AgentScope AgentId
  | ToolboxScope ToolboxId
  | ConversationScope ConversationId
  | TurnScope TurnId
  | ToolCallScope ToolCallId
  deriving (Eq, Ord, Show)
```

### Resource Components
```haskell
newtype ResourceInfo = ResourceInfo
  { resourceId :: ResourceId
  , resourceScope :: ResourceScope
  , resourceType :: ResourceType
  , resourceCreatedAt :: UTCTime
  , resourceLastAccessed :: TVar UTCTime
  } deriving (Show)

instance Component ResourceInfo where
  componentId _ = ComponentTypeId 10

data ResourceType
  = SqliteResource SqliteConfig
  | LuaResource LuaConfig
  | HttpManagerResource HttpManagerConfig
  | ProcessResource ProcessConfig
  deriving (Show)

data ResourceHandle = ResourceHandle
  { handleId :: ResourceId
  , handleCleanup :: IO ()
  , handleAccess :: forall a. (ResourceAccessor -> IO a) -> IO a
  }

newtype ResourceRegistry = ResourceRegistry
  { registryHandles :: TVar (HashMap ResourceId ResourceHandle)
  , registryCounter :: TVar Int
  }

-- Context for resource operations
newtype ResourceContext = ResourceContext
  { contextScope :: [ScopeLevel]
  , contextRegistry :: ResourceRegistry
  }
```

## Tradeoffs

1. **Explicit Cleanup vs Finalizers**: Explicit cleanup (`handleCleanup`) is used for predictable resource management. Finalizers are non-deterministic.

2. **Resource Sharing**: Shared resources use the same ResourceId. The registry tracks implicitly via cleanup - if two entities share a resource, only one should register cleanup.

3. **Scope Checking**: Runtime scope checking adds overhead but ensures correctness. Consider caching scope lookups.
