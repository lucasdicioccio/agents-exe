### 💬 User Query

# Scope Lua sandbox per tool-call for maximum isolation and safety

Depends-on: 
Base-branch: gh-lua-per-tool-call
Final: false

## Summary

Change the Lua sandbox resource scoping from **per-conversation** to **per-tool-call**. Each invocation of the Lua toolbox should create a fresh, isolated Lua state that is destroyed after execution completes.

## Motivation

### Current Problem

The current Lua toolbox maintains persistent state across tool calls within the same conversation:

```lua
-- Tool Call 1: Sets global state
users = { "alice", "bob" }

-- Tool Call 2: Can see and modify Tool Call 1's globals
table.insert(users, "charlie")
```

**Issues with persistent state:**
1. **Non-determinism**: LLM can't reason about what state exists from previous calls
2. **Memory leaks**: Accumulating globals in long conversations  
3. **Security**: One compromised tool call could poison state for subsequent calls
4. **Debugging difficulty**: Hard to reproduce issues when state depends on call history
5. **Deadlock risk**: When Lua calls agent recursively via tool portal, both agents contend for same `TMVar` lock

### Proposed Solution

Each `lua_{name}_execute` tool call gets a **fresh Lua state** that is destroyed immediately after execution:

```haskell
executeScript script = do
    bracket Lua.newstate Lua.close $ \lstate -> do
        configureSandbox lstate
        registerStandardModules lstate portal
        runScript lstate script
```

## Benefits

| Benefit | Explanation |
|---------|-------------|
| **Maximum isolation** | Each call starts from identical initial state |
| **Memory bounded** | No accumulation of Lua heap across calls |
| **Secure by default** | No way for call N to influence call N+1 |
| **No deadlocks** | No shared lock between parent and recursive sub-agent |
| **Deterministic** | Same inputs always produce same behavior |
| **Easier testing** | Each call is independently testable |

## Trade-offs

| Trade-off | Assessment |
|-----------|------------|
| **Initialization overhead** | ~1-3ms per call (acceptable) |
| **No implicit persistence** | Use SQLite for explicit persistence needs |
| **Module re-registration** | Fast, only happens per-call |

## Implementation

### Key Changes

1. **Remove persistent state from `LuaToolbox`**
   - Remove `toolboxLuaState :: Lua.State`
   - Remove `toolboxLock :: MVar ()`
   - Create fresh state per call using `bracket`

2. **Simplify resource management**
   - Lua no longer needs to be a managed resource in the OS
   - Remove or simplify `System.Agents.OS.Resources.Lua`
   - `ExclusiveAccess` pattern no longer needed

3. **Update `executeScriptWithPortal`**
   - Use `bracket` pattern for guaranteed cleanup
   - Initialize modules fresh for each call

### Code Sketch

```haskell
-- BEFORE: Persistent state with lock
data Toolbox = Toolbox
    { toolboxLuaState :: Lua.State
    , toolboxLock :: MVar ()
    , toolboxConfig :: LuaToolboxDescription
    }

-- AFTER: Stateless, fresh per call
data Toolbox = Toolbox
    { toolboxConfig :: LuaToolboxDescription
    , toolboxMakePortal :: IO ToolPortal
    }

executeScriptWithPortal tracer toolbox script portal = do
    bracket 
        (initializeFreshState tracer toolbox portal)
        (destroyState tracer)
        (\lstate -> executeScriptInternal lstate script)
```

## Migration for Scripts

Scripts that relied on persistence must use SQLite:

```lua
-- BEFORE (won't work after change):
-- Call 1: users = { "alice" }
-- Call 2: return users[1]  -- Returns "alice"

-- AFTER (explicit persistence):
local json = require("json")
local tools = require("tools")

-- Save
tools.call("sqlite_memory_query", {
    sql = "INSERT INTO state VALUES ('users', '" .. json.encode(users) .. "')"
})

-- Load
local result = tools.call("sqlite_memory_query", {
    sql = "SELECT value FROM state WHERE key = 'users'"
})
users = json.decode(result.result_txt)
```
