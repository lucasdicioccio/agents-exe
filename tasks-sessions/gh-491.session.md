### 💬 User Query

# Add session introspection tools to SystemToolbox for cross-session analysis

Depends-on: 
Base-branch: main
Final: false

## Summary

Add tools to the SystemToolbox that allow agent sessions to introspect other sessions from the session store and search index. This enables powerful cross-session analysis, pattern recognition, and context sharing between related conversations.

## Motivation

Currently, an agent session cannot easily introspect other sessions. This limits the ability to:
- Learn from previous similar conversations
- Build cumulative knowledge across session boundaries
- Enable parent/child session relationships for complex multi-step workflows
- Search for relevant context from past sessions

## Proposed Solution

Add a new `SessionIntrospectionCapability` to the SystemToolbox with the following capabilities:

### 1. Session Listing (`list_sessions`)
List sessions with optional filtering:
- **Scope options**:
  - `parent_sessions` - Sessions that are parents of the current session (via `forkedFromSessionId`)
  - `child_sessions` - Sessions forked from the current session
  - `all_sessions` - All accessible sessions in the session store
- **Filters**: by date range, agent slug, tool usage patterns
- **Sorting**: by modification time, turn count, relevance

### 2. Session Search (`search_sessions`)
Full-text search across session content using the existing FTS5 index:
- Search in user queries, LLM responses, tool outputs
- Filters: date range, agent, tools used
- Returns: session metadata with match previews and relevance scores
- Integrates with existing `Session.Search.Query` infrastructure

### 3. Session Reading (`read_session`)
Retrieve detailed session information:
- Full session content (turns, tool calls, responses)
- Session statistics (turn counts, token usage, byte counts)
- Trajectory signals (interaction, execution, environment signals)
- Option to return raw JSON or markdown-formatted output
- Content selection: all turns, last N turns, specific turn range

### 4. Session Statistics (`get_session_stats`)
Quick access to aggregated statistics without full content:
- Turn counts (user/LLM/total)
- Tool usage summary (which tools, how many times)
- Token and byte usage
- Signal metrics (informativeness score, failure count, etc.)
- Duration and timing information

## Scope Configuration

The scope of accessible sessions should be configurable in `SystemToolboxDescription`:

```haskell
data SessionIntrospectionScope
  = ScopeParentsOnly    -- Can only see parent sessions (ancestors)
  | ScopeChildrenOnly   -- Can only see child sessions (descendants)
  | ScopeSubtree        -- Parents + current + children
  | ScopeAll            -- All sessions (requires explicit opt-in)
  deriving (Show, Eq)
```

## Implementation Details

### New Types

Add to `System.Agents.Base`:
```haskell
data SessionIntrospectionCapability
  = SessionIntrospectionList
  | SessionIntrospectionSearch
  | SessionIntrospectionRead
  | SessionIntrospectionStats
  deriving (Show, Ord, Eq, Generic)

data SystemToolCapability = 
  -- ... existing capabilities ...
  | SystemToolSessionIntrospection SessionIntrospectionCapability
```

### Update Module: `System.Agents.Tools.SystemToolbox`

Following the pattern of `System.Agents.Tools.SystemToolbox`, create new branches to configure, trace, and execute the SystemToolbox

1. **Initialization**: Takes `SessionIntrospectionToolboxDescription` with scope and capabilities
2. **Query Execution**: Provides functions for each capability
3. **Access Control**: Enforces scope restrictions based on `forkedFromSessionId` relationships
4. **Result Formatting**: JSON output suitable for LLM consumption

### Session Lineage Tracking

Leverage existing `System.Agents.OS.Conversation.Lineage` for tracking parent/child relationships:
- Use `forkedFromSessionId` from `Session` type
- Build ancestry/descendant trees from session store

### Integration Points

1. **SessionStore** (`System.Agents.SessionStore`):
   - `listSessions` - enumerate available sessions
   - `readSession` - load session content
   
2. **Session.Search.Index** (`System.Agents.Session.Search.Index`):
   - `executeSearchWithOptions` - search across sessions
   
3. **SessionPrint** (`System.Agents.SessionPrint`):
   - `calculateStatistics` - compute session statistics
   - `formatSessionAsMarkdown` - format for readability

## Security and Privacy Considerations

1. **Scope Enforcement**: Strictly enforce the configured scope to prevent unauthorized access
2. **Content Filtering**: Option to censor sensitive tool outputs (similar to `sessionEditCensorToolCalls`)
3. **Rate Limiting**: Consider query rate limits to prevent abuse
4. **Audit Trail**: Log introspection queries for security review

## Tradeoffs

1. **Performance**: Cross-session queries may be slow for large session stores
   - Use the FTS5 index for search (already exists)
   - Cache session metadata
   - Lazy loading of session content

2. **Memory**: Loading multiple sessions could consume significant memory
   - Implement pagination for list results
   - Streaming for large session content
   - Limit maximum sessions returned

3. **Complexity**: Adding session relationships increases system complexity
   - Keep parent/child tracking simple (single parent, many children)
   - Leverage existing `forkedFromSessionId` field

4. **Scope vs. Flexibility**: More restrictive scopes are safer but less useful
   - Default to `ScopeSubtree` for balanced security/usability
   - Require explicit opt-in for `ScopeAll`
