# Define ToolExecutionContext type for passing session metadata to tools

## Overview

Create a new type `ToolExecutionContext` that encapsulates all session-related metadata that tools might need access to. This context will be passed to tool execution but will NOT be exposed to the LLM.

## Data to Include

The context should contain:
- `SessionId` - the current session identifier
- `ConversationId` - the current conversation identifier  
- `Maybe Session` - optionally the full serialized session (for tools that need full context)
- `TurnId` - the current turn identifier
- `Maybe AgentId` - the agent identifier

## Type Definition Location

Create in `src/System/Agents/Tools/Context.hs`:

```haskell
module System.Agents.Tools.Context where

data ToolExecutionContext = ToolExecutionContext
    { ctxSessionId :: SessionId
    , ctxConversationId :: ConversationId
    , ctxTurnId :: TurnId
    , ctxAgentId :: Maybe AgentId
    , ctxFullSession :: Maybe Session  -- JSON serializable full session
    }
```

## Design Decisions

**Tradeoffs considered:**

1. **Generic `Aeson.Value` vs structured record**: A structured record is preferred because:
   - Type safety - callers know exactly what context is available
   - Better documentation - fields are self-documenting
   - Easier evolution - adding fields is backward compatible if done carefully

2. **`Maybe Session` vs always including session**: Making it optional allows:
   - Performance optimization when full session isn't needed
   - Avoiding circular dependencies if Session depends on tools
   - Smaller context for simple tools

3. **Separate module vs adding to existing**: New module allows:
   - Clean import hierarchy (Tools.Base can import Context)
   - Potential future expansion without bloating existing modules
