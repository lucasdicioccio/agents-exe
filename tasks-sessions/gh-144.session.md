# Implement `toolbox mcp` command for direct MCP toolbox mapping without agent indirection

## Overview

Implement `agents-exe toolbox mcp` command that directly maps a toolbox over MCP without requiring an agent as an indirection layer. This allows tools from an agent's toolbox to be exposed as an MCP server without the current `turnAgentRuntimeIntoIOTool` conversion.

## Motivation

Currently, exposing tools via MCP requires:
1. Loading an agent
2. Converting the agent to an IO tool via `OneShotTool.turnAgentRuntimeIntoIOTool`
3. Running the MCP server with this indirection

This adds complexity and overhead. Users should be able to directly expose a toolbox (bash tools, OpenAPI tools, PostgREST tools) as an MCP server without going through an agent.

## Current Architecture

```haskell
-- Current flow in McpServer.multiAgentsServer:
-- Agent -> turnAgentRuntimeIntoIOTool -> IOTool -> MCP Server
```

The current implementation in `McpServer.multiAgentsServer` takes `AgentTree.Props` and uses `OneShotTool.turnAgentRuntimeIntoIOTool` to convert agents into tools.

## Proposed Interface

```bash
# Start MCP server exposing all tools from an agent's toolbox (no LLM agent)
agents-exe toolbox mcp --agent-file agent.json --stdio

# Expose only bash tools from a directory
agents-exe toolbox mcp --tool-dir ./tools --stdio

# Expose tools from multiple sources
agents-exe toolbox mcp \
  --agent-file agent.json \
  --mcp-toolbox "filesystem:./fs-mcp.json" \
  --stdio

# With logging
agents-exe toolbox mcp --agent-file agent.json --stdio --log-level debug
```

## Tradeoffs

### Pros
- Lower overhead (no LLM calls for tool routing)
- Simpler mental model (tools directly exposed)
- Faster tool execution (no agent indirection)
- Can expose tools without configuring an LLM

### Cons
- Loses agent-level features (tool selection logic, retries, etc.)
- MCP tools calling other MCP tools may need special handling
- Less consistent with the agent-centric architecture

## Summary

Closes #144

Implemented `agents-exe toolbox mcp` command for direct MCP toolbox mapping without agent indirection.

### Changes Made

1. **New Module: `System.Agents.Tools.McpServer`** - A complete direct MCP server implementation that:
   - Exposes tools from multiple sources (agent configs, directories, MCP servers, OpenAPI, PostgREST)
   - Executes tools directly without LLM agent routing
   - Supports configurable name mapping strategies (KeepPrefixedNames, StripPrefixes, UseOriginalNames)
   - Uses stdio transport for MCP protocol
   - Reuses existing MCP protocol handling from `System.Agents.MCP.Server`

2. **Updated `app/Main.hs`**:
   - Added new `ToolboxMcp` command with CLI parser
   - Added `McpToolboxOptions` data type
   - Added handlers for the new command
   - Fixed `parseListToolsOptions` to use correct parser for `ToolsList.ToolSource`

3. **Updated `agents.cabal`**:
   - Added `System.Agents.Tools.McpServer` to exposed modules

### Usage Examples

```bash
# Start MCP server exposing all tools from an agent's toolbox (no LLM agent)
agents-exe toolbox mcp --agent-file agent.json --stdio

# Expose only bash tools from a directory
agents-exe toolbox mcp --tool-dir ./tools --stdio

# Expose tools from multiple sources
agents-exe toolbox mcp \
  --agent-file agent.json \
  --mcp-toolbox "filesystem:./fs-mcp.json" \
  --stdio

# With logging
agents-exe toolbox mcp --agent-file agent.json --stdio --log-level debug
```

### Key Benefits

- **Lower overhead**: No LLM calls for tool routing
- **Simpler deployment**: No LLM API keys needed for tool-only servers
- **Direct tool execution**: Bypasses agent-level features like retries
- **Tool aggregation**: Combine tools from multiple sources into one MCP server
- **Consistent with existing architecture**: Reuses existing MCP protocol handling and tool registration systems
