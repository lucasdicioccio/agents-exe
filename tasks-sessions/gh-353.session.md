# OS Model: Migration Path - Runtime Compatibility Layer

Depends-on: #5
Base-branch: gh-os-architecture
Final: false

## Overview

Design the migration path from the current Runtime-per-agent architecture to the new OS model. This provides a compatibility layer that allows gradual migration of existing code.

## Background

The current architecture has:
- `Runtime` per agent with tools in `TVar`
- `Agent` type with step function
- `OneShot` and `TUI` using `Runtime` directly

We need a bridge that:
- Maintains backward compatibility
- Allows gradual migration
- Provides escape hatches for performance-critical paths

## Types to Define

### Compatibility Bridge
```haskell
-- | Bridge from old Runtime interface to new OS
newtype RuntimeBridge = RuntimeBridge
  { bridgeAgentId :: AgentId
  , bridgeOS :: OS
  }

-- Old Runtime interface (typeclass for compatibility)
class Monad m => AgentRuntime m where
  listTools :: m [ToolRegistration]
  callTool :: Text -> Value -> m UserToolResponse
  getTracer :: m (Tracer IO Trace)

-- New OS-backed implementation
instance AgentRuntime (ReaderT RuntimeBridge IO) where
  listTools = do
    bridge <- ask
    tools <- runOSM (bridgeOS bridge) $ 
             getAgentTools (bridgeAgentId bridge)
    return tools
  
  callTool name input = do
    bridge <- ask
    result <- runOSM (bridgeOS bridge) $ do
      let agentId = bridgeAgentId bridge
      -- Find or create conversation for this agent
      convId <- startConversation agentId (ConversationConfig Nothing mempty)
      turnId <- startTurn convId
      executeToolCall turnId name input
    return $ callResultToUserToolResponse result
```

### Migration State
```haskell
newtype MigrationState = MigrationState
  { migrationPhase :: MigrationPhase
  , migrationCompatRuntimes :: TVar (Map AgentId RuntimeBridge)
  }

data MigrationPhase
  = PhaseOldOnly        -- Only old Runtime
  | PhaseDual           -- Both coexist
  | PhaseNewOnly        -- Only OS
  deriving (Show, Eq)

newtype MigrationConfig = MigrationConfig
  { migrationEnableOS :: Bool
  , migrationEnableCompat :: Bool
  , migrationLogPath :: Maybe FilePath
  }

-- Initialization with migration support
initializeWithMigration :: MigrationConfig -> IO (Either String (Either Runtime OS))
-- Returns Left Runtime if using old system, Right OS if using new
```

## Migration Guide

### Phase 1: Dual Mode (Current)
```haskell
-- Old code continues to work
result <- newRuntime slug announce tracer apiKey model ...
```

### Phase 2: Gradual Migration
```haskell
-- New code uses OS directly
os <- initializeOS config
createAgent os agentConfig
```

### Phase 3: Full Migration
```haskell
-- Compatibility layer removed
-- All code uses OS directly
```

## Tradeoffs

1. **Migration Duration**: Longer migration maintains two systems. Dual-mode operation should be kept brief (1-2 releases).

2. **Performance Overhead**: Compatibility layer adds overhead. Document escape hatches for performance-critical paths.

3. **Testing Burden**: Dual-mode requires testing both paths. Use feature flags to control which path is active.

4. **API Surface**: Compatibility layer may not support all OS features. New features may be OS-only.
