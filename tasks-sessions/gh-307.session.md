### 💬 User Query

# Stateful resource management of individual toolboxes, and individual agents

As we add more builtin toolboxes, a problem we face is managing access to resources held by tool executions.

Tool executions require resources like:
- sqlite file handles
- lua interpreters
- http-managers 
- process file descriptors

While a single conversation thread may keep long-running access to resources, the case of concurrent conversations within the TUI is interesting:
- multiple agents may need the same toolboxes (e.g., a same sqlite db being called concurrently)
- some resources may benefit from sharing across toolboxes (e.g., http-managers re-using established connexions)

In these case, what is important is to clarify:
- what is the lifecycle of the resource being allocated
  * per program (resource is like a global)
  * per agent (the agent hold some resource)
  * per toolbox (the toolbox instantiated possibly multiple times by the agent)
  * per conversation (multiple invocations along the same conversation share an isolation box)
  * per turn (multiple tools in a same turn share an isolation box)
  * per toolcall (each tool has a separate isolation box)
 
I believe we'd need significant rework of the core structures into something more flat than we have today.

The core "Operating System" would have:
- a list of agent descriptions (loaded from disk, but maybe in the future from elsewhere)
- a list of toolboxes descriptions (loaded from disk, but maybe in the future from elsewhere)
- a list of toolboxes runtimes (to share or not resources across agents/sessions/turns/toolcalls)
- mappings from agents to toolboxes (possibly, instantiation by adding arguments specific to the agent)
- graph structure between agents, tools to build trees, dags, and cycles of agents
- some threads to manage ongoing conversations including
   - mapping of sessions to agents and toolboxes
   - ongoing conversation threads
   - the lineage between conversations
   - ongoing tool-calls (e.g., to support asynchronous calls)

What we'll want in the end is so that:
- OneShot is a trivial execution of one conversation
- TUI is a simple wrapper on top of the "Operating system"
- It opens the door to serving agents-exe as a web-API 
- We can store sessions and toolcalls (requested, resulted) in a RDBMS

For tracing and computing we need:
- we need to trace the lineage of everything very thorough (an agent calling another agent tool in a conversation from a discussion generated from a toolcall should be traceable)
- we need to be able to accumulate computations and state across two axes: as we compute through a stack of recursive/nested calls and along conversation turns

The general architectural pattern I would like to follow for the core data type is the one I take as example in https://dicioccio.fr/how-i-write-purescript-apps.html#the-cheap-entity-component-system .

### Key Design Decisions

1. **ECS Pattern**: Uses flat, table-like storage with Entity IDs and Components, inspired by the PureScript ECS article
2. **Resource Scopes**: Six lifecycle levels (Program, Agent, Toolbox, Conversation, Turn, ToolCall)
3. **STM-based Concurrency**: Composable transactions for shared resource access
4. **Gradual Migration**: Compatibility layer allows coexistence of old and new systems
5. **Persistence Layer**: RDBMS storage for durability and querying

### Module Structure

```
System.Agents.OS.Core          - Entity, Component, World
System.Agents.OS.Resources     - Resource lifecycle, scopes
System.Agents.OS.Concurrent    - STM locks, access patterns
System.Agents.OS.Conversation  - Turn, lineage, messages
System.Agents.OS               - Main OS monad
System.Agents.OS.Compat        - Runtime compatibility
System.Agents.OS.Persistence   - RDBMS storage
System.Agents.OS.Interfaces.*  - TUI, OneShot, API
```

### Tradeoffs Highlighted

- **Type Erasure vs Safety**: Using `TVar Any` for heterogeneous storage
- **Explicit Cleanup vs Finalizers**: Predictable resource management
- **STM vs MVar**: Composability vs slight performance overhead
- **Migration Duration**: Dual-mode adds complexity but ensures smooth transition
- **JSON vs Normalized**: Flexibility vs queryability in persistence
