# Tool Hooks System

Depends-on: 
Base-branch: gh-501
Final: false

## Overview
Implement a hooks system for tools supporting pre/post execution hooks, middleware, and conditional behavior modification.

## Motivation
Advanced users need to customize tool behavior without modifying tool source code. Hooks enable:
- **Customization**: Modify tool inputs/outputs
- **Integration**: Connect tools to external systems
- **Validation**: Add custom validation logic
- **Logging**: Enhanced audit trails
- **Transformation**: Automatic data transformations

## Proposed Solution
Implement a hooks system where users can register functions to run before/after tool execution.

### Hook Types
1. **Pre-execution**: Modify arguments before tool runs
2. **Post-execution**: Modify results after tool completes
3. **Error hook**: Handle or transform errors
4. **Condition**: Only run hook when condition is met

### Configuration Example
```yaml
tool_hooks:
  - tool: bash
    pre_exec:
      - name: log_command
        action: log_info
    post_exec:
      - name: validate_output
        condition: "exit_code == 0"
        action: validate_json
```

### Implementation Sketch
```haskell
-- Hook definition
data ToolHook = ToolHook {
    hookType :: HookType,  -- PreExec | PostExec | OnError
    condition :: Maybe Condition,
    action :: HookAction
}

-- Hook registry
type HookRegistry = Map ToolName [ToolHook]

-- Execute tool with hooks
runWithHooks :: HookRegistry -> ToolName -> Arguments -> IO Result
```

   - `HookDefinition` - Individual hook configuration
   - `HookConditionConfig` - Serializable condition types
   - `HookActionConfig` - Serializable action types
   - `loadHookConfigFile` - YAML/JSON file loading
   - `hookRegistryFromConfig` - Convert config to runtime registry
   - `validateHookConfig` - Configuration validation

3. **`agents.cabal`** - Updated to include new modules

### Features Implemented:

- ✅ Pre-execution hooks can modify tool arguments
- ✅ Post-execution hooks can modify tool results
- ✅ Error hooks can intercept and handle failures
- ✅ Conditional execution of hooks based on context (exit code, tool name, arguments)
- ✅ Hook registry can be configured via config file (YAML/JSON)
- ✅ Multiple hooks can be chained for a single tool
- ✅ Composable conditions (And, Or, Not)

### Example Configuration:

```yaml
tool_hooks:
  - tool: bash
    pre_exec:
      - name: log_command
        action:

          level: info
      - name: validate_args

          key: command
          value: rm
        action:
          type: validate
          message: "rm command requires confirmation"
    post_exec:
      - name: validate_output
        condition:
          type: exit_code_is
          value: 0
        action:
          type: transform
          operation: parse_json
```
